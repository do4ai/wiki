# 02. 백엔드 프로젝트 세팅 트랙
이 장의 목표는 백엔드를 빨리 띄우는 것이 아니라, 뒤에서 장바구니, 주문, 결제, 운영자 권한이 자연스럽게 얹히는 서버의 바닥을 먼저 만드는 것입니다. 실전 프로젝트에서 서버가 자주 망가지는 이유는 초기 구조 없이 endpoint만 늘리기 때문입니다. 그래서 이 장에서는 FastAPI 앱을 하나 띄우는 데서 끝나지 않고, 애플리케이션 계층, 설정 계층, 데이터베이스 연결, 테스트 시작점까지 함께 세웁니다.

# 이 장의 목표
- 이커머스 백엔드가 무엇을 책임지고 무엇을 프론트엔드에 넘겨야 하는지 설명할 수 있습니다.
- FastAPI 기반 서버를 라우터, 서비스, 저장소, 설정 계층으로 나눠 시작할 수 있습니다.
- 인증, 권한, 주문 상태 전이를 고려한 데이터 구조의 출발점을 만들 수 있습니다.
- 이후 DDD 기반 도메인 구현이 가능한 서버 골조를 세울 수 있습니다.

# 먼저 결정해야 하는 것
백엔드 세팅에서 가장 먼저 해야 할 일은 기술보다 경계를 정하는 것입니다.

- API는 어떤 역할을 노출하는가
- 서비스 계층은 어디서 비즈니스 규칙을 수행하는가
- 저장소 계층은 어디까지 데이터 영속화만 담당하는가
- 인증과 세션은 어떤 방식으로 다룰 것인가
- 운영자 권한은 일반 사용자와 어떻게 분리할 것인가

이 질문에 답하지 않으면 서버는 금방 "라우터에 모든 것이 몰린 구조"가 됩니다.

# 이 장의 결과물
- `server/api`에서 route 단위 진입점을 관리합니다.
- `server/shared` 또는 도메인 패키지에서 서비스와 저장소를 분리합니다.
- 설정과 환경변수 로딩 계층을 둡니다.
- 데이터베이스 연결과 세션 생명주기를 분명히 둡니다.
- 헬스체크와 기본 테스트가 존재합니다.

# 어떤 기술을 선택하는가
이 교재의 백엔드 기본 선택은 다음과 같습니다.

- FastAPI
- SQLAlchemy
- SQLite로 시작하는 영속화
- pytest
- ruff

여기서 중요한 것은 "이 기술이 유행인가"가 아니라, 빠른 실습과 구조 설명에 적합한가입니다. SQLite로 시작하는 이유는 설치 부담을 낮추면서도 영속화와 migration 사고방식을 설명할 수 있기 때문입니다. 이후 더 큰 데이터베이스로 옮길 수 있도록 구조를 먼저 분리해 둡니다.

# 폴더를 어떻게 나누는가
초기 백엔드는 보통 다음처럼 책임별로 나누는 것이 안전합니다.

- `api/`: 요청과 응답, 인증 진입점, HTTP 경계
- `domain/` 또는 `shared/service.py`: 비즈니스 규칙, 상태 전이, 계산 규칙
- `infra/` 또는 `repository`: DB 연결, 저장소 구현, 외부 연동
- `config/`: 환경변수, 설정 객체, 실행 환경 분기
- `tests/`: 기능 시나리오와 기본 회귀 확인

실전에서는 이 나눔이 나중에 더 세분화될 수 있지만, 초기에 이 구분이 없으면 거의 반드시 라우터에 규칙이 쌓이기 시작합니다.

# 테스트를 왜 바로 넣어야 하는가
이 장부터 테스트를 함께 두는 이유는 간단합니다. 주문과 결제는 화면보다 API와 상태 전이가 먼저 신뢰를 얻어야 하기 때문입니다. 그래서 다음 정도는 시작점에서 확인 가능해야 합니다.

- 헬스체크 endpoint가 동작하는가
- 상품 조회 endpoint가 최소 응답을 돌려주는가
- 인증 실패와 성공이 구분되는가
- 테스트 DB가 매 실행마다 재현 가능한가

# 이 장에서 흔히 하는 실수
- FastAPI 앱 하나에 모든 라우터와 규칙을 몰아 넣는다.
- ORM 모델을 곧바로 도메인 모델처럼 다룬다.
- 설정값을 코드에 직접 하드코딩한다.
- 테스트 없이 CRUD부터 늘린다.
- 운영자와 일반 사용자 권한 구분을 뒤로 미룬다.

# 이 장에서 확인해야 할 체크리스트
- 서버가 독립적으로 실행되는가
- 환경변수 로딩과 설정 객체가 존재하는가
- DB 연결과 세션 생명주기가 분리되어 있는가
- 최소 테스트가 실행되는가
- 인증과 권한 분리를 확장 가능한 구조로 둘 준비가 되었는가

# 다음 장으로 넘어가기 전에
다음 장에서는 인프라 프로젝트 세팅을 다룹니다. 따라서 이 장이 끝났을 때 백엔드는 기능이 많을 필요는 없지만, "실행 환경 안에서 재현 가능하게 돌아가는 서버"여야 합니다. 실전에서는 좋은 서버 코드만으로는 제품이 되지 않습니다. 실행과 연결이 함께 준비되어야 합니다.
