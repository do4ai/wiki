# 4. AI 상담 챗봇 대화와 화면 기획
앞 페이지까지에서 우리는 storefront와 admin의 화면 구조를 정리했습니다. 이제 마지막으로 AI 상담 챗봇을 어떤 사용자 경험으로 붙일지 정해야 합니다. 이번 교재에서 챗봇은 부가 장식이 아닙니다. 상품 문의, 주문 문의, FAQ 응답, 상담 전환까지 담당하는 지원 흐름의 일부입니다.

많은 프로젝트에서 챗봇은 마지막에 작은 말풍선 위젯처럼 붙습니다. 하지만 그렇게 만들면 실제로는 도움을 주지 못합니다. 사용자가 어디서 질문을 던지는지, 어떤 질문은 바로 답해도 되는지, 어떤 질문은 사람이나 운영 흐름으로 넘겨야 하는지 설계하지 않으면 챗봇은 "있는 것처럼 보이는 기능"이 됩니다.

이번 페이지의 목적은 챗봇을 제품 경험 안에 자연스럽게 넣는 것입니다. 어떤 화면에서 어떤 맥락으로 열리고, 어떤 질문을 처리하고, 어떤 순간에 상담 전환을 제안해야 하는지를 먼저 정해야 뒤의 AI/백엔드 구현이 흔들리지 않습니다.

# 왜 챗봇 UX를 따로 설계해야 하는가
챗봇은 일반 페이지와 다르게 "질문 -> 해석 -> 응답 -> 다음 행동"이라는 별도 흐름을 가집니다. 그래서 일반 화면처럼 버튼과 카드만 놓는다고 끝나지 않습니다.

설계 없이 바로 붙이면 보통 아래 문제가 생깁니다.

- 모든 화면에서 같은 말풍선이 뜨는데 실제로는 어디서 써야 하는지 모르겠다
- 상품 질문과 주문 질문을 구분하지 않아 엉뚱한 답변을 한다
- FAQ는 잘 답하지만, 해결이 안 될 때 다음 행동이 없다
- 챗봇이 할 수 없는 질문까지 받다가 신뢰를 잃는다

즉 챗봇 UX의 핵심은 많이 답하는 것이 아니라, 적절한 질문을 적절한 시점에 처리하고, 처리 못하는 경우에도 사용자를 버리지 않는 것입니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 챗봇이 노출될 화면
- 질문 유형별 대화 흐름
- FAQ/상품/주문 질문의 처리 방식
- 핸드오프 또는 운영 전환 기준
- 채팅 UI의 최소 구성

이 문서가 정리되면 다음 단계에서 챗봇 백엔드와 프론트 패널을 구현하기 쉬워집니다.

# 1. 이번 교재에서 챗봇이 해야 하는 일
이번 MVP에서 챗봇의 역할은 명확하게 제한합니다.

- 상품 관련 질문에 답한다
- 주문 관련 기본 질문에 답한다
- FAQ를 안내한다
- 해결이 어려우면 다음 행동을 제시한다

중요한 점은 이 챗봇이 범용 비서가 아니라는 것입니다. 이커머스 상담 assistant이므로, 답변 범위는 서비스 도메인 안으로 제한하는 편이 맞습니다.

# 2. 챗봇이 열려야 하는 화면
챗봇은 어디서나 뜨게 만들 수 있지만, 이번 교재에서는 "질문이 자연스럽게 발생하는 화면"에 집중하는 것이 좋습니다.

## 우선 노출 위치
- 상품 상세 화면
- 장바구니 또는 주문/결제 화면
- 주문 상세 또는 주문 확인 화면

## 이유
- 상품 상세: 상품 정보, 옵션, 가격, 재고 관련 질문이 많다
- 주문/결제: 배송, 결제, 정책, 실패 처리 질문이 많다
- 주문 확인: 주문 상태와 후속 조치 질문이 많다

## 굳이 강하게 노출하지 않아도 되는 곳
- 홈 첫 화면
- 운영자 admin 화면

챗봇은 항상 전면에 있는 것보다, 사용자가 막히는 지점에서 자연스럽게 여는 편이 더 좋습니다.

# 3. 질문 유형을 먼저 나눈다
챗봇 설계의 첫 단계는 질문 유형을 나누는 것입니다. 이번 교재에서는 최소한 아래 세 가지로 나누면 충분합니다.

## 3.1 상품 질문
예:

- 이 상품은 어떤 용도인가요?
- 가격이 얼마인가요?
- 옵션이 있나요?
- 다른 상품과 차이가 뭔가요?

### 처리 방식
- 상품 상세 정보 기반 응답
- 짧고 명확한 설명
- 필요하면 상세 화면 정보와 연결

## 3.2 주문 질문
예:

- 주문이 완료됐나요?
- 결제가 실패했는데 어떻게 하나요?
- 배송은 언제 되나요?

### 처리 방식
- 주문 상태 기반 응답
- 현재 상태 설명
- 재시도 또는 확인 경로 안내

## 3.3 FAQ/정책 질문
예:

- 환불은 어떻게 되나요?
- 배송비는 얼마인가요?
- 회원가입이 필요한가요?

### 처리 방식
- 정책성 답변
- 정리된 FAQ 응답
- 필요한 경우 관련 화면 이동 안내

# 4. 대화 흐름의 기본 구조
이번 교재의 챗봇 대화는 아래 구조를 기본으로 생각하면 됩니다.

1. 사용자가 질문한다
2. 챗봇이 질문 유형을 파악한다
3. 적절한 정보 소스에서 답을 만든다
4. 답변과 함께 다음 행동을 제안한다
5. 해결이 안 되면 핸드오프를 제안한다

중요한 점은 "답변만 주고 끝내지 않는다"는 것입니다. 좋은 챗봇 UX는 다음 행동을 같이 보여 줍니다.

# 5. 상품 질문 대화 흐름
상품 질문은 주로 상품 상세 화면에서 발생합니다.

## 목적
- 사용자의 구매 결정을 돕는다
- 상세 설명을 보조한다

## 기본 흐름
1. 사용자가 상품 관련 질문 입력
2. 챗봇이 현재 상품 컨텍스트를 기준으로 응답
3. 필요하면 상세 설명/가격/정책 정보를 짧게 요약
4. 마지막에 장바구니 담기 또는 추가 확인을 유도

## 좋은 응답 예시 방향
- 길게 설명하기보다 핵심 정보 위주
- 현재 보고 있는 상품을 기준으로 응답
- 다른 화면으로 이동할 필요가 있으면 CTA 제안

# 6. 주문 질문 대화 흐름
주문 질문은 주문 상세, 주문 완료, 결제 실패 지점에서 많이 나옵니다.

## 목적
- 사용자의 불안을 줄인다
- 상태를 해석해 준다

## 기본 흐름
1. 사용자가 주문/결제 관련 질문 입력
2. 챗봇이 주문 맥락을 읽는다
3. 현재 상태를 쉬운 말로 설명한다
4. 다음 행동을 제안한다

## 예시
- 결제 실패 시: 실패 상태 설명 + 재시도/주문 확인 경로 제안
- 주문 완료 시: 주문 접수 상태 설명 + 주문 상세로 이동 제안

주문 질문은 정보 정확성이 중요하므로, 지나친 생성형 자유도를 주기보다 상태 기반 답변이 적합합니다.

# 7. FAQ/정책 질문 대화 흐름
FAQ 질문은 챗봇이 가장 안정적으로 처리할 수 있는 영역입니다.

## 목적
- 반복 질문을 줄인다
- 기본 정책 안내를 자동화한다

## 기본 흐름
1. 사용자가 정책 질문 입력
2. 챗봇이 FAQ 문서 또는 정책 답변에서 응답 생성
3. 필요한 경우 관련 화면 또는 다음 행동 제안

## 설계 원칙
- 지나치게 창의적으로 답하지 않는다
- 정책 문구는 일관되게 유지한다

# 8. 핸드오프는 언제 필요한가
챗봇 UX에서 가장 중요한 요소 중 하나는 "언제 넘길 것인가"입니다. 챗봇이 해결할 수 없는 문제를 끝까지 붙잡으면 사용자 신뢰가 떨어집니다.

이번 교재에서는 아래 경우에 핸드오프를 제안하는 것이 자연스럽습니다.

- 질문이 도메인 범위를 벗어난다
- 주문 상태 해석만으로 해결되지 않는다
- 결제 실패 원인을 확정적으로 설명할 수 없다
- 동일 질문이 반복되는데 해결되지 않는다

즉 핸드오프는 실패가 아니라, 안전장치입니다.

# 9. 핸드오프 UX는 어떻게 보여야 하는가
핸드오프는 단순히 "모르겠습니다"가 되어선 안 됩니다. 다음 행동이 보여야 합니다.

## 보여 줄 것
- 해결이 어렵다는 짧은 설명
- 주문 상세 보기 또는 관련 화면 이동 버튼
- 운영 문의 또는 고객센터 안내 문구

## 피해야 할 것
- 무조건 긴 사과문
- 아무 행동도 없는 종료 메시지

좋은 핸드오프는 챗봇 경험을 끊는 것이 아니라, 사람 또는 다른 화면으로 자연스럽게 잇는 것입니다.

# 10. 채팅 UI의 최소 구성
이번 교재의 UI는 과하게 화려할 필요 없습니다. 대신 사용성이 분명해야 합니다.

## 최소 요소
- 질문 입력창
- 대화 메시지 리스트
- 빠른 질문 또는 추천 질문
- 로딩 상태
- 오류 상태
- 핸드오프 CTA

## UI 원칙
- 응답은 너무 길지 않게
- 중요한 상태 정보는 눈에 띄게
- 버튼형 다음 행동을 함께 제시

챗봇은 텍스트만 잘 보여도 충분히 쓸 만할 수 있습니다. 핵심은 말풍선 디자인보다 흐름입니다.

# 11. 추천 질문은 어디에 필요한가
추천 질문은 사용자가 "무엇을 물어볼 수 있는지" 이해하지 못할 때 도움이 됩니다.

이번 교재에서 추천 질문은 특히 아래 화면에서 유용합니다.

- 상품 상세
- 주문 완료/상세
- 결제 실패 상황

## 예시
- 이 상품은 어떤 특징이 있나요?
- 배송은 언제 시작되나요?
- 결제가 실패했는데 어떻게 해야 하나요?

추천 질문은 챗봇 사용 장벽을 낮추는 가장 쉬운 도구입니다.

# 12. 실패 상태와 오류 상태
챗봇도 일반 화면처럼 실패 상태가 필요합니다.

## 예시
- 응답 생성 실패
- 주문 정보를 읽을 수 없음
- 정책 답변을 찾지 못함

## 보여 줄 것
- 다시 시도 버튼
- 직접 확인할 수 있는 화면 이동
- 필요 시 핸드오프 제안

이번 교재에서는 "모든 상황에 답한다"보다 "답하지 못할 때도 사용자에게 길을 준다"가 더 중요합니다.

# 13. 실습용 챗봇 플로우 초안
이번 교재를 따라 하는 학습자는 최소한 아래 흐름을 먼저 적어 두면 충분합니다.

| 질문 유형 | 주 화면 | 기본 응답 | 다음 행동 |
| --- | --- | --- | --- |
| 상품 질문 | 상품 상세 | 상품 설명/가격/옵션 안내 | 장바구니 담기 또는 상세 확인 |
| 주문 질문 | 주문 상세/완료 | 상태 설명 | 주문 상세 이동 또는 재시도 |
| FAQ 질문 | 결제/정책 관련 화면 | 정책 안내 | 관련 화면 이동 |
| 해결 불가 | 모든 핵심 화면 | 제한 안내 | 핸드오프 또는 운영 문의 |

# 14. 자주 하는 실수
- 챗봇을 모든 화면에서 똑같이 띄운다
- 상품 질문, 주문 질문, 정책 질문을 구분하지 않는다
- 답하지 못할 때 다음 행동을 주지 않는다
- 지나치게 긴 생성형 답변을 허용한다
- 챗봇을 UI 위젯으로만 보고 실제 서비스 흐름과 연결하지 않는다

이 다섯 가지 실수는 챗봇을 기능 데모로만 남기고 실제 지원 도구로 못 쓰게 만듭니다.

# 정리
이 페이지에서 우리는 AI 상담 챗봇이 어디서 열리고, 어떤 질문을 처리하고, 언제 핸드오프를 해야 하는지 정리했습니다. 이제 storefront, admin, assistant surface까지 화면 기획의 기본 골격이 모두 잡혔습니다. 다음 단계에서는 이 설계를 실제로 구현하기 위한 개발 환경과 인프라 구성을 시작합니다.
