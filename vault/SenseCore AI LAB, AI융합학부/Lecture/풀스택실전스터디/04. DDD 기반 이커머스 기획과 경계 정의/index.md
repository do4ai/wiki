# 04. DDD 기반 이커머스 기획과 경계 정의
이 장은 이 교재의 방향을 결정하는 핵심 장입니다. 앞의 세 장에서 프론트엔드, 백엔드, 인프라가 각각 설 준비를 마쳤다면, 이제는 무엇을 어떤 경계로 만들 것인지 결정해야 합니다. 이커머스는 화면 수가 많은 서비스처럼 보이지만, 실제로는 상품, 사용자, 장바구니, 주문, 결제, 운영이라는 서로 다른 문제들이 만나는 시스템입니다. 이 장에서는 그 문제들을 DDD 관점으로 잘라서 뒤의 구현 순서를 흔들리지 않게 만듭니다.

# 이 장의 목표
- 이커머스 서비스를 bounded context 단위로 나눌 수 있습니다.
- aggregate, entity, value object, application service가 어디서 필요한지 설명할 수 있습니다.
- 기능명세와 화면설계를 도메인 규칙으로 번역할 수 있습니다.
- 뒤 장에서 구현할 순서를 도메인 의존성 기준으로 정리할 수 있습니다.

# 먼저 정리해야 하는 bounded context
이 교재에서는 최소한 다음 경계를 구분합니다.

- `catalog`: 상품, 가격, 노출 정보
- `identity`: 사용자, 인증, 역할
- `cart`: 사용자별 담기 상태와 수량 변경
- `order`: 주문 생성, 주문 항목, 총액, 상태 전이
- `payment`: 결제 시도와 결제 확인
- `admin`: 운영 관찰과 관리 기능

중요한 점은 이 경계를 파일 폴더로만 나누는 것이 아니라, 각 경계가 어떤 규칙을 소유하는지 분명히 하는 것입니다.

# 무엇을 하나의 aggregate로 볼 것인가
예를 들어 장바구니와 주문은 겉보기에는 비슷해 보여도 다릅니다.

- 장바구니는 계속 수정되는 임시 상태입니다.
- 주문은 한 시점의 스냅샷이며, 생성 이후에는 상태 전이가 중요합니다.

이 차이를 무시하면 장바구니와 주문을 같은 테이블과 같은 규칙으로 다루게 되고, 결제와 주문 확정에서 반드시 문제가 생깁니다. 그래서 이 장에서는 "무엇이 언제 확정되는가"를 중심으로 aggregate를 나눕니다.

# 기능명세를 어떻게 읽는가
실전에서는 명세 문서가 곧바로 코드가 되지 않습니다. 먼저 다음 질문으로 번역해야 합니다.

- 누가 이 기능을 사용하는가
- 어떤 입력이 필요하고 무엇이 검증되어야 하는가
- 상태는 어디서 바뀌는가
- 실패하면 어떤 이유로 거절되는가
- 다른 컨텍스트와 어떤 데이터가 오가는가

이 과정을 거쳐야 route와 entity와 테이블과 화면이 같은 방향을 보게 됩니다.

# 이 장의 결과물
- bounded context 목록과 책임 정의
- 핵심 유스케이스 목록
- aggregate 후보와 상태 전이 메모
- 제품 용어집 초안
- 구현 우선순위 표

# 이 장에서 흔히 하는 실수
- 화면 메뉴를 그대로 도메인 경계라고 착각한다.
- CRUD 표만 만들고 상태 전이를 정의하지 않는다.
- 운영자 기능을 나중에 붙일 별도 화면 정도로만 본다.
- 결제를 주문 내부 세부사항으로만 다루고 별도 흐름으로 보지 않는다.
- 용어를 팀마다 다르게 쓰는데도 정리하지 않는다.

# 이 장에서 확인해야 할 체크리스트
- 상품, 사용자, 장바구니, 주문, 결제, 운영자 기능의 경계가 분명한가
- 주문과 장바구니를 다른 상태 모델로 설명할 수 있는가
- 사용자용 기능과 관리자용 기능의 권한 경계가 분명한가
- 명세 문서를 유스케이스와 규칙으로 번역했는가
- 뒤 장의 구현 순서가 도메인 의존성에 맞는가

다음 장에서는 이제 실제 백엔드 구현 순서를 따라갑니다. 따라서 이 장이 끝났을 때 가장 중요한 결과는 문서 분량이 아니라 경계의 선명함입니다. 구현이 느려 보여도 이 단계가 정확하면 뒤 장의 속도와 안정성이 크게 달라집니다.
