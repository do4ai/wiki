# 09. 최종 통합, 검증, 확장
이 장은 프로젝트를 끝내는 법을 다룹니다. 많은 실습은 기능이 돌아가는 순간 종료되지만, 실전 제품은 전체 흐름을 다시 묶어 보고, 실패 흐름을 점검하고, 다음 확장 방향까지 판단해야 비로소 정본에 가까워집니다. 이 장에서는 지금까지 만든 storefront, admin, backend, infrastructure를 하나의 서비스 시나리오로 다시 엮습니다.

# 이 장의 목표
- 전체 사용자 시나리오를 처음부터 끝까지 점검할 수 있습니다.
- 구매자와 관리자 기준의 검증 항목을 따로 정리할 수 있습니다.
- 현재 구조의 한계와 다음 확장 포인트를 설명할 수 있습니다.
- 더 큰 규모의 프로젝트로 가기 위한 다음 단계를 설계할 수 있습니다.

# 무엇을 검증해야 하는가
- 회원가입과 로그인
- 상품 조회와 장바구니 담기
- 주문 생성과 결제 확인
- 주문 이후 상태 조회
- 운영자 로그인과 대시보드 확인
- 주문과 결제 상태 관찰

성공 흐름만 보는 것으로는 부족합니다. 실패 흐름도 반드시 확인해야 합니다.

- 인증 없이 접근했을 때
- 잘못된 주문 요청을 보냈을 때
- 결제 확인이 실패했을 때
- 운영자 권한이 없는 사용자가 admin에 접근했을 때

# 확장 방향을 어떻게 보는가
이 장에서는 다음과 같은 확장 질문을 다룹니다.

- SQLite에서 MySQL 또는 PostgreSQL로 언제 옮길 것인가
- 외부 결제 게이트웨이를 어떻게 붙일 것인가
- 재고와 쿠폰과 배송을 어떤 컨텍스트로 추가할 것인가
- 관리자 기능을 어디까지 세분화할 것인가
- 모놀리식 구조를 유지할지 더 큰 분리로 갈지 언제 판단할 것인가

# 이 장의 결과물
- 최종 사용자 시나리오 검증표
- 실패 흐름 점검표
- 기술 부채와 다음 개선 목록
- 확장 우선순위 메모

# 이 장에서 흔히 하는 실수
- 성공 흐름 데모만 보고 프로젝트를 끝냈다고 판단한다.
- 관리자 흐름 검증을 생략한다.
- 데이터베이스 확장과 결제 확장 포인트를 문서화하지 않는다.
- 현재 구조의 한계를 인정하지 않는다.
- 다음 프로젝트로 넘길 교훈을 남기지 않는다.

# 이 장에서 확인해야 할 체크리스트
- 구매자와 관리자 시나리오가 모두 통과하는가
- 실패 흐름의 에러 메시지와 상태가 이해 가능한가
- 현재 구조의 한계가 정리되어 있는가
- 다음 확장 우선순위가 있는가
- 이 프로젝트를 기반으로 더 큰 서비스로 갈 수 있는 설명이 가능한가

# 이 책을 마치며
이 장을 마치면 학생은 기능 몇 개를 붙여 본 사람이 아니라, 하나의 이커머스 제품을 처음 세팅하는 단계부터 검증과 확장까지 설명할 수 있는 사람이 되어야 합니다. 이 교재의 목적은 특정 프레임워크 문법을 많이 익히는 것이 아니라, 실제 제품을 끝까지 가져가는 순서를 몸에 남기는 것입니다.
