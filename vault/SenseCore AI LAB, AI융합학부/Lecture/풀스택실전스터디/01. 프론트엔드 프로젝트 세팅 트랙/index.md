# 01. 프론트엔드 프로젝트 세팅 트랙
이 장의 목표는 화면을 예쁘게 만드는 것이 아니라, 이후의 상품 조회, 인증, 장바구니, 주문, 운영자 기능을 견딜 수 있는 프론트엔드 바닥을 먼저 세우는 것입니다. 실전 프로젝트는 대개 화면 구현부터 급하게 시작하지만, 그렇게 출발하면 나중에 API 연결과 권한 분리와 운영자 화면을 붙이는 순간 구조가 바로 무너지기 쉽습니다. 그래서 이 장에서는 가장 먼저 storefront와 admin을 왜 분리해야 하는지 이해하고, 각각을 어떤 책임으로 시작해야 하는지부터 정리합니다.

# 이 장의 목표
- 구매자용 화면과 운영자용 화면이 왜 별도 앱이어야 하는지 설명할 수 있습니다.
- React + TypeScript 기반 프론트엔드 프로젝트를 이커머스 제품 기준으로 시작할 수 있습니다.
- 라우팅, 폴더 구조, 환경변수, API 접근 계층, 공통 UI 레이어를 초기 단계에서 어떻게 나눠야 하는지 이해할 수 있습니다.
- 이후 백엔드와 연결될 것을 전제로 화면 구조를 먼저 설계할 수 있습니다.

# 이 장을 읽는 방법
- 이 장은 코드보다 먼저 책임과 구조를 봅니다.
- 각 절에서 먼저 어떤 문제를 막기 위한 설계인지 이해한 뒤에 설정과 파일 구조를 따라갑니다.
- 여기서 만든 프론트엔드 구조는 뒤 장에서 백엔드와 실제로 연결될 것이므로, 임시 샘플이라고 생각하지 말고 제품의 첫 골조라고 생각해야 합니다.

# 먼저 결정해야 하는 것
프론트엔드 세팅에서 가장 먼저 해야 할 결정은 "앱을 몇 개로 나눌 것인가"입니다. 이 교재에서는 처음부터 두 개로 나눕니다.

- `storefront`: 구매자가 상품을 보고, 로그인하고, 장바구니에 담고, 주문하는 앱
- `admin`: 운영자가 상품, 주문, 결제 시도, 운영 상태를 보는 앱

이렇게 나누는 이유는 단순히 URL이 다르기 때문이 아닙니다.

- 구매자와 운영자는 보는 정보가 다릅니다.
- 구매자와 운영자는 권한이 다릅니다.
- 구매자와 운영자는 화면의 목적이 다릅니다.
- 나중에 배포 단위나 접근 정책이 달라질 가능성도 큽니다.

`palcar` 같은 규모의 프로젝트가 커질수록 이 차이는 더 커집니다. 그래서 초반에 하나의 거대한 프론트엔드로 시작했다가 나중에 admin을 뜯어내는 방식보다, 처음부터 두 개의 앱을 같은 monorepo 안에 두고 역할을 분리하는 쪽이 훨씬 안전합니다.

# 이 장의 결과물
이 장을 끝내면 최소한 아래 구조가 잡혀 있어야 합니다.

```text
client/
├─ storefront/
│  ├─ src/
│  │  ├─ app/
│  │  ├─ lib/
│  │  ├─ pages/ 또는 route 단위 화면
│  │  └─ styles/
│  ├─ package.json
│  └─ vite.config.ts
└─ admin/
   ├─ src/
   │  ├─ app/
   │  ├─ lib/
   │  ├─ pages/ 또는 route 단위 화면
   │  └─ styles/
   ├─ package.json
   └─ vite.config.ts
```

중요한 점은 지금 이 시점에서 화면 수가 많은지가 아닙니다. 중요한 것은 이후의 기능이 들어와도 파일이 어디로 가야 하는지가 분명한 상태를 만드는 것입니다.

# 어떤 기술을 선택하는가
이 교재는 프론트엔드 기본 스택으로 다음을 둡니다.

- React
- TypeScript
- Vite
- React Router
- fetch 기반 API 접근 계층

이 선택은 최신 유행만 따른 것이 아닙니다.

- React는 컴포넌트와 상태를 제품 구조 안에서 설명하기 쉽습니다.
- TypeScript는 API 계약과 화면 데이터를 일찍부터 구조화하기 좋습니다.
- Vite는 초기 셋업과 로컬 개발 속도가 빠릅니다.
- React Router는 구매자 흐름과 운영자 흐름의 route 경계를 설명하기에 적절합니다.

이 장의 목적은 "왜 이 스택이 지금 이 제품에 맞는가"를 이해하는 것입니다. 더 중요한 것은 선택한 스택 위에서 구조를 무너지지 않게 시작하는 것입니다.

# 폴더를 어떻게 나누는가
실전 프로젝트에서 프론트엔드가 빠르게 망가지는 대표적인 이유는 모든 것을 `components/` 하나에 쌓기 시작하기 때문입니다. 이 교재에서는 처음부터 역할에 따라 나눕니다.

## `app/`
- 앱 셸, 라우팅, 전역 provider, 인증 가드, layout처럼 앱 전체를 조직하는 코드

## `lib/`
- API 클라이언트, 타입 정의, 공통 유틸리티, 세션 저장소 같은 재사용 로직

## 화면 계층
- 실제 페이지 또는 route 단위 UI
- 예: 홈, 상품 목록, 장바구니, 로그인, 주문 상태, admin overview

## `styles/`
- 앱의 공통 스타일 레이어
- 디자인 토큰, 배경, typography, 공통 surface 규칙

이 구분은 단순한 취향이 아니라, 나중에 어떤 코드가 어느 층의 책임인지 빠르게 찾기 위해 필요합니다.

# storefront를 어떻게 시작하는가
구매자용 앱은 처음부터 모든 화면을 다 만들지 않습니다. 먼저 다음 질문에 답할 수 있는 구조로 시작합니다.

1. 비회원은 무엇을 볼 수 있는가?
2. 로그인한 사용자는 무엇이 달라지는가?
3. 상품 데이터는 어디서 읽는가?
4. 장바구니 상태는 서버 상태인가, 로컬 상태인가?
5. 주문은 어떤 route와 어떤 상태 전이 위에서 처리되는가?

초기 storefront에서 필요한 최소 화면은 보통 다음과 같습니다.

- 홈 또는 상품 목록
- 로그인/회원가입 진입점
- 장바구니
- 체크아웃 또는 주문 생성 진입점
- 최근 주문 상태 확인

이 장에서는 이 화면들을 모두 완성하지 않아도 됩니다. 대신 각각이 어디에 들어가야 하고, 어떤 API를 기다리는지 설명 가능한 상태를 만들어야 합니다.

# admin을 어떻게 시작하는가
운영자용 앱은 storefront를 복사해서 색만 바꾸는 앱이 아닙니다. 운영자 앱은 읽는 정보와 행동의 목적이 다릅니다.

운영자 앱에서 먼저 준비해야 할 것은 다음입니다.

- 로그인 화면
- 보호된 route
- overview dashboard
- 상품 목록
- 주문 목록
- 결제 시도 또는 운영 이벤트 목록

여기서 핵심은 "운영자에게 필요한 것은 구매 흐름이 아니라 관찰과 판단을 위한 구조"라는 점입니다. 그래서 admin 앱은 처음부터 dashboard shell, nav, table, status badge 같은 운영 UI의 뼈대를 갖고 시작하는 편이 좋습니다.

# 환경변수와 API 기준점을 어떻게 둬야 하는가
프론트엔드가 백엔드 없이도 돌아가야 한다고 생각하면 나중에 API 연결에서 크게 흔들립니다. 반대로 백엔드가 아직 다 구현되지 않았다고 프론트엔드 작업을 멈추면 진행이 막힙니다. 그래서 이 장에서는 다음 원칙을 둡니다.

- API base URL은 환경변수로 분리한다.
- API 호출은 화면 안에 직접 흩뿌리지 않고 `lib/api.ts` 같은 접근 계층으로 모은다.
- 응답 타입은 가능한 한 초기에 정의한다.
- 아직 없는 endpoint는 mock 또는 fallback으로 잠시 견디되, 나중에 실제 endpoint로 바뀔 자리를 분명히 둔다.

이 방식 덕분에 프론트엔드는 백엔드 구현과 병렬로 진행될 수 있습니다. 동시에 나중에 실제 API 계약이 확정됐을 때 수정 범위를 좁게 유지할 수 있습니다.

# 라우팅은 왜 일찍 잡아야 하는가
실전 프로젝트에서 route는 단순한 URL이 아닙니다. route는 사용자 흐름과 책임의 경계입니다.

예를 들어 storefront에서는 다음과 같은 흐름이 생깁니다.

- `/` 또는 상품 목록
- `/login`
- `/cart`
- `/checkout`
- `/orders`

admin에서는 다음과 같은 흐름이 생깁니다.

- `/login`
- `/overview`
- `/products`
- `/orders`
- `/payments`

이 route들을 일찍 잡는 이유는 화면 수를 늘리기 위해서가 아니라, 어떤 기능이 어느 흐름에 속하는지를 빨리 고정하기 위해서입니다. route가 늦게 정해지면 UI와 API와 상태가 한데 엉켜 버립니다.

# 이 장에서 흔히 하는 실수
- storefront와 admin을 한 앱으로 시작한다.
- API 호출을 각 화면 컴포넌트 안에 직접 작성한다.
- 타입 정의 없이 응답 shape를 즉석에서 소비한다.
- 폴더 구조를 기능이 아니라 파일 종류 기준으로만 쌓아 둔다.
- 인증 전과 인증 후 화면의 차이를 구조로 나누지 않는다.

이 실수들은 초반에는 빨라 보여도, 주문과 권한과 운영자 기능이 붙는 순간 거의 반드시 다시 뜯게 만듭니다.

# 이 장에서 확인해야 할 체크리스트
- `storefront`와 `admin`이 별도 앱으로 존재하는가
- 각 앱이 독립적으로 실행되는가
- 공통적으로 `app`, `lib`, `styles` 계층이 존재하는가
- API base URL이 환경변수나 설정 레이어로 분리되어 있는가
- route 구조가 제품 흐름 기준으로 설명 가능한가
- 이후 backend/auth/cart/order/admin 기능이 들어갈 위치가 분명한가

# 다음 장으로 넘어가기 전에
다음 장에서는 백엔드 프로젝트를 세팅합니다. 따라서 이 장이 끝났을 때 프론트엔드는 "화면을 많이 만든 상태"일 필요는 없지만, "어떤 API가 들어오면 어디에 연결할지 분명한 상태"여야 합니다. 즉 프론트엔드는 먼저 완성되는 것이 아니라, 백엔드가 붙을 수 있는 골격을 먼저 얻는 것입니다.

이 장을 제대로 마치면 학생은 단순한 UI 클론을 만드는 사람이 아니라, 앞으로 인증과 주문과 운영을 받아낼 수 있는 프론트엔드 제품 구조를 먼저 세울 수 있게 됩니다.
