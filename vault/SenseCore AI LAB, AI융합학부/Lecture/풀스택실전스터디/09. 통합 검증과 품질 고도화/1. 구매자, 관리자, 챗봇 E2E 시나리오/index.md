# 1. 구매자, 관리자, 챗봇 E2E 시나리오
지금까지 우리는 서비스 기획, DDD 모델링, 화면 기획, 인프라, 백엔드와 프론트 골격, 카탈로그, 인증, 장바구니, 주문, 결제, 운영자 화면, AI 상담 챗봇까지 단계적으로 만들었습니다. 이제부터는 각 기능이 따로 동작하는지보다, 하나의 이커머스 서비스로 이어지는지를 검증합니다.

이번 페이지의 목표는 테스트 도구 사용법을 나열하는 것이 아닙니다. 사용자가 실제로 상품을 사고, 운영자가 그 주문을 확인하고, 챗봇이 같은 주문과 상품 맥락을 안전하게 안내하는지 끝에서 끝까지 확인하는 시나리오를 만드는 것입니다.

즉 이번 장은 "기능이 구현되어 있다"를 "서비스로 믿고 실행할 수 있다"로 바꾸는 단계입니다.

# 왜 E2E 시나리오가 필요한가
단위 테스트와 API 테스트는 필요합니다. 하지만 풀스택 서비스에서는 그것만으로 충분하지 않습니다.

- API는 성공하는데 프론트 상태가 갱신되지 않을 수 있다
- 프론트는 보이는데 실제 주문 데이터가 저장되지 않을 수 있다
- 주문은 생성되는데 admin 대시보드에 반영되지 않을 수 있다
- 챗봇은 답하지만 권한 없는 주문을 설명할 수 있다
- 상품 가격 변경이 과거 주문 snapshot을 깨뜨릴 수 있다

E2E 시나리오는 이런 연결 문제를 찾기 위한 검증입니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 어떤 사용자 흐름을 최우선 E2E로 둘 것인가
- 구매자, 관리자, 챗봇 시나리오는 어떻게 연결되는가
- 성공 흐름과 실패 흐름은 어떻게 나눌 것인가
- 테스트 데이터는 어떤 상태에서 시작해야 하는가
- 자동화할 것과 수동 점검으로 남길 것은 무엇인가
- 통과 기준은 어떤 관찰 결과로 판단할 것인가

# 1. E2E는 화면 순서가 아니라 업무 흐름 기준으로 잡는다
E2E를 화면 목록대로 만들면 테스트가 많아 보이지만 실제 위험을 잘 잡지 못합니다. 이번 교재에서는 업무 흐름 기준으로 묶습니다.

- 상품을 찾는다
- 로그인한다
- 장바구니에 담는다
- 주문을 생성한다
- 결제를 확인한다
- 주문 완료를 본다
- admin에서 주문을 확인한다
- 챗봇에게 주문 상태를 묻는다

이 흐름 하나가 통과하면 구매자, 백엔드, DB, admin, 챗봇 연결이 함께 검증됩니다.

# 2. 테스트 데이터는 어떻게 준비하는가
E2E는 시작 상태가 안정적이어야 합니다. 매번 다른 데이터에서 시작하면 실패 원인을 판단하기 어렵습니다.

초기 seed는 아래 정도가 필요합니다.

- 판매 중인 상품 2개
- 품절 상품 1개
- 비노출 상품 1개
- buyer 사용자 1명
- admin 사용자 1명
- 기본 배송/환불 정책 문서
- 빈 장바구니 상태

테스트가 주문을 생성한다면, 실행 전 장바구니와 주문 데이터를 정리하거나 테스트 전용 사용자로 분리하는 것이 좋습니다.

# 3. 핵심 E2E 1: 구매 성공 흐름
첫 번째 핵심 시나리오는 정상 구매 흐름입니다.

검증 순서는 아래와 같습니다.

1. buyer로 로그인한다
2. 상품 목록을 연다
3. 판매 중인 상품 상세로 이동한다
4. 장바구니에 담는다
5. 장바구니에서 수량과 합계를 확인한다
6. 주문을 생성한다
7. mock payment를 성공 처리한다
8. 주문 완료 화면을 확인한다

통과 기준은 단순히 화면이 넘어가는 것이 아닙니다. 주문 ID, 주문 상태, 결제 상태, 금액 합계가 예상과 일치해야 합니다.

# 4. 핵심 E2E 2: admin 주문 확인 흐름
두 번째 시나리오는 구매자가 만든 주문이 운영자 화면에 반영되는지 확인합니다.

검증 순서는 아래와 같습니다.

1. admin으로 로그인한다
2. 대시보드 요약을 확인한다
3. 주문 목록에서 방금 생성한 주문을 찾는다
4. 주문 상세로 이동한다
5. 주문 항목, 총액, 주문 상태, 결제 상태를 확인한다
6. 고객 지원 맥락 placeholder가 있는지 확인한다

통과 기준은 storefront에서 생성한 주문과 admin에서 보는 주문이 같은 데이터라는 점입니다. 프론트별로 별도 mock 데이터를 쓰고 있으면 이 시나리오에서 바로 드러납니다.

# 5. 핵심 E2E 3: 챗봇 주문 상태 질문
세 번째 시나리오는 AI 상담 챗봇이 실제 주문 맥락을 안전하게 사용하는지 확인합니다.

검증 순서는 아래와 같습니다.

1. buyer로 주문 상세 화면에 들어간다
2. 챗봇을 연다
3. "내 주문은 지금 어떤 상태인가요?"라고 질문한다
4. 챗봇 응답에 주문 상태와 결제 상태가 반영되는지 확인한다
5. 답변에 운영자 연결이 필요한지 여부가 자연스럽게 표시되는지 확인한다

통과 기준은 챗봇이 근거 없는 일반 답변을 하지 않고, 현재 사용자의 주문 상태를 권한 검사 후 반영하는 것입니다.

# 6. 핵심 E2E 4: 권한 없는 주문 접근 차단
챗봇과 admin을 붙이면 권한 검증이 더 중요해집니다.

검증 순서는 아래와 같습니다.

1. buyer A로 주문을 만든다
2. buyer B로 로그인한다
3. buyer A의 주문 ID를 직접 URL이나 챗봇 질문으로 시도한다
4. 주문 상세 접근이 막히는지 확인한다
5. 챗봇도 해당 주문 정보를 설명하지 않는지 확인한다

통과 기준은 백엔드가 권한을 막는 것입니다. 프론트 라우팅에서만 막는 것은 충분하지 않습니다.

# 7. 핵심 E2E 5: 상품 상태 변경과 storefront 반영
상품 운영 도구가 있다면 admin 변경이 구매자 화면에 반영되는지도 검증해야 합니다.

검증 순서는 아래와 같습니다.

1. admin으로 상품 운영 화면에 들어간다
2. 특정 상품의 노출 상태를 끈다
3. storefront 상품 목록에서 해당 상품이 사라졌는지 확인한다
4. 판매 상태를 중지한다
5. 상품 상세 또는 장바구니 추가가 막히는지 확인한다
6. 이미 생성된 주문의 상품 snapshot은 변하지 않는지 확인한다

통과 기준은 현재 상품 상태와 과거 주문 snapshot이 분리되어 동작하는 것입니다.

# 8. 실패 흐름은 무엇을 먼저 검증하는가
성공 흐름만 검증하면 실제 서비스 품질을 알기 어렵습니다. 최소한 아래 실패 흐름은 포함합니다.

- 비회원이 장바구니 이후 주문 생성을 시도한다
- 품절 상품을 장바구니에 담으려 한다
- 결제 실패 후 주문 상태와 UI 안내를 확인한다
- admin 권한이 없는 사용자가 admin에 접근한다
- 챗봇이 근거 없는 질문을 받는다
- LLM 호출 실패 시 재시도 또는 handoff가 보인다

실패 흐름의 통과 기준은 "에러가 난다"가 아니라, 사용자가 다음 행동을 이해할 수 있어야 한다는 점입니다.

# 9. 자동화 우선순위는 어떻게 정하는가
모든 시나리오를 처음부터 자동화할 필요는 없습니다. 하지만 반복될수록 위험한 흐름은 자동화해야 합니다.

자동화 우선순위는 아래 순서가 적절합니다.

1. 구매 성공 흐름
2. admin 주문 확인 흐름
3. 권한 없는 주문 접근 차단
4. 결제 실패 흐름
5. 상품 상태 변경 반영
6. 챗봇 주문 상태 질문

챗봇은 LLM 응답이 변동될 수 있으므로, 답변 문장 전체보다 구조화 응답과 핵심 상태 포함 여부를 검증하는 편이 좋습니다.

# 10. 테스트 도구는 무엇을 쓰는가
프론트 E2E는 Playwright 같은 브라우저 테스트 도구가 적합합니다. 백엔드 API 흐름은 pytest 기반 통합 테스트로 검증할 수 있습니다.

이번 교재의 권장 구조는 아래와 같습니다.

- backend integration test: 도메인 규칙과 API 계약 검증
- frontend E2E: 실제 화면 전이와 사용자 행동 검증
- smoke test: 로컬 또는 배포 환경에서 최소 생존 확인
- manual checklist: 자동화가 어려운 UX와 운영 판단 확인

중요한 것은 도구 이름이 아니라, 검증 대상이 겹치지 않게 역할을 나누는 것입니다.

# 11. E2E 테스트는 어디에 두는가
모노레포에서는 테스트 위치도 명확히 해야 합니다.

예시 구조는 아래와 같습니다.

- `server/tests/integration`: 백엔드 통합 테스트
- `client/storefront/e2e`: 구매자 E2E
- `client/admin/e2e`: 관리자 E2E
- `tests/e2e`: 여러 앱을 동시에 다루는 cross-app E2E
- `sdd/03_verify`: 사람이 읽는 검증 시나리오와 결과 기록

코드로 실행되는 테스트와 사람이 읽는 검증 기록을 섞지 않는 것이 좋습니다.

# 12. 관찰 기준은 어떻게 남기는가
E2E는 통과/실패만으로 끝내지 말고, 무엇을 봤는지 남겨야 합니다.

각 시나리오는 아래 항목을 기록합니다.

- 사전 조건
- 실행 단계
- 기대 결과
- 관찰한 화면 또는 API 상태
- 실패 시 원인 후보
- 자동화 여부

이 기록이 있어야 다음 사람이 실패를 재현할 수 있습니다.

# 13. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 구매 성공 E2E 시나리오가 정의되어 있다
- admin 주문 확인 시나리오가 정의되어 있다
- 챗봇 주문 상태 질문 시나리오가 정의되어 있다
- 권한 실패와 결제 실패 흐름이 포함되어 있다
- 테스트 데이터와 초기화 규칙이 정리되어 있다
- 자동화 우선순위가 정해져 있다
- 사람이 읽는 검증 기록 위치가 정해져 있다

즉 이제 기능별 개발이 아니라 제품 단위 검증으로 넘어갈 준비가 된 상태입니다.

# 14. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- 구매 성공 E2E 테스트
- admin 주문 확인 E2E 테스트
- 권한 실패 통합 테스트
- 결제 실패 테스트
- 상품 상태 변경 반영 테스트
- 챗봇 구조화 응답 테스트
- E2E seed/reset 스크립트
- 검증 결과 문서

이 산출물이 있어야 다음 페이지에서 성능, 보안, 권한 점검을 더 좁고 정확하게 수행할 수 있습니다.

# 15. 자주 하는 실수
- 화면 하나가 열리는지만 보고 E2E라고 부른다
- 프론트와 백엔드가 같은 테스트 데이터를 보지 않는다
- 주문 생성 후 admin 반영을 확인하지 않는다
- 챗봇 답변 문장 전체를 고정해 불안정한 테스트를 만든다
- 권한 실패를 프론트 라우팅으로만 검증한다
- 실패 흐름의 사용자 안내를 확인하지 않는다
- 테스트 실행 전 데이터 초기화를 하지 않는다

이 일곱 가지 실수는 테스트가 있어도 실제 제품 품질을 보장하지 못하게 만듭니다.

# 정리
이번 페이지에서는 구매자, 관리자, AI 상담 챗봇을 하나의 이커머스 서비스 흐름으로 검증하는 E2E 시나리오를 정리했습니다. 핵심은 화면별 테스트가 아니라, 주문이 생성되고 운영자 화면에 반영되며 챗봇이 같은 맥락을 안전하게 설명하는지 확인하는 것입니다.
