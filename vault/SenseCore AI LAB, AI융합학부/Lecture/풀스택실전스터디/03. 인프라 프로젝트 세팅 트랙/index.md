# 03. 인프라 프로젝트 세팅 트랙
이 장의 목표는 배포를 나중 일로 미루지 않고, 실행 가능성과 재현 가능성을 프로젝트의 시작점에 넣는 것입니다. 이커머스 프로젝트는 프론트엔드와 백엔드가 각각 잘 만들어져도, 환경변수와 네트워크와 데이터 저장 위치가 정리되지 않으면 팀 전체가 흔들립니다. 그래서 이 장에서는 로컬 개발 환경과 실행 환경을 코드와 같은 수준으로 관리하는 습관을 먼저 만듭니다.

# 이 장의 목표
- 프론트엔드와 백엔드를 함께 띄우는 실행 환경을 설계할 수 있습니다.
- 환경변수, 포트, 데이터 저장 위치, 네트워크 경로를 분명히 정의할 수 있습니다.
- 컨테이너와 실행 스크립트가 왜 필요한지 설명할 수 있습니다.
- 나중에 CI와 배포로 이어질 기반 파일을 준비할 수 있습니다.

# 먼저 이해해야 하는 것
인프라는 배포 담당자만 보는 영역이 아닙니다. 프로젝트를 만드는 모든 사람이 이해해야 하는 기본 바닥입니다.

- 프론트엔드는 어디를 API 서버로 바라보는가
- 백엔드는 어떤 DB를 어디에 저장하는가
- 컨테이너를 쓰면 무엇이 재현 가능한가
- `.env` 파일은 어떤 책임을 가지는가
- 개발용 실행과 운영용 실행은 무엇이 다른가

이 질문을 초반에 정리하면 뒤에서 "내 컴퓨터에서는 되는데 왜 안 되지" 같은 시간이 크게 줄어듭니다.

# 이 장의 결과물
- `compose.yml` 또는 동등한 실행 정의 파일
- 프론트엔드와 백엔드를 띄우는 스크립트
- 개발용 환경변수 샘플
- 데이터 파일과 로그 위치에 대한 규칙
- 헬스체크와 기본 부팅 검증 방법

# 어떤 기준으로 설계하는가
이 교재는 처음부터 거대한 인프라를 요구하지 않습니다. 대신 다음 원칙을 둡니다.

- 로컬에서 누구나 같은 방식으로 부팅할 수 있어야 한다.
- 프론트엔드와 백엔드의 연결 경로가 명시적이어야 한다.
- 상태가 남는 경로와 사라지는 경로를 구분해야 한다.
- 나중에 CI와 배포로 옮겨도 이름과 구조가 크게 바뀌지 않아야 한다.

# 어떤 파일이 필요한가
- `compose.yml`: 서비스 연결의 기준점
- `scripts/dev/*.sh`: 반복 실행 작업 자동화
- `.env.example`: 필수 환경변수 문서화
- 서비스별 README 또는 실행 가이드

이 파일들은 편의를 위한 옵션이 아니라, 팀 전체의 공통 약속입니다.

# 이 장에서 흔히 하는 실수
- 인프라를 "나중에 배포할 때" 정리하려고 미룬다.
- 포트와 API 경로를 코드 곳곳에 하드코딩한다.
- 데이터 파일 위치를 개발자 개인 환경에 맡긴다.
- 프론트와 백엔드를 각자 수동으로만 실행하게 둔다.
- 환경변수 문서화를 생략한다.

# 이 장에서 확인해야 할 체크리스트
- 한 명이 아니라 누구나 같은 명령으로 프로젝트를 띄울 수 있는가
- 프론트엔드와 백엔드의 연결 경로가 명확한가
- 개발용 환경변수와 운영용 환경변수를 구분할 준비가 되어 있는가
- 저장 데이터와 임시 데이터의 위치가 분명한가
- 헬스체크나 기본 부팅 검증 방법이 존재하는가

# 다음 장으로 넘어가기 전에
다음 장에서는 DDD 기반으로 이커머스 도메인을 본격적으로 기획합니다. 따라서 이 장이 끝났을 때 인프라는 거창할 필요는 없지만, 이후의 도메인 설계와 구현 결과를 실제로 실행하고 검증할 수 있는 바닥이어야 합니다. 구현을 쌓아도 돌릴 수 없으면 그 프로젝트는 아직 제품이 아닙니다.
