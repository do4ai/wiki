# 05. 백엔드 개발 순서 따라가기
이 장은 이 교재의 구현 중심축입니다. 여기서는 기능을 생각나는 대로 만들지 않습니다. 이커머스 서비스가 실제로 서기 위해 필요한 순서를 고정하고, 그 순서대로 백엔드 규칙을 두껍게 쌓아 갑니다. 좋은 백엔드는 endpoint 개수가 많은 서버가 아니라, 도메인 상태 전이가 일관된 서버입니다.

# 이 장의 목표
- 백엔드 기능을 의존성 순서대로 구현할 수 있습니다.
- catalog, identity, cart, order, payment 흐름을 하나의 제품 서사로 연결할 수 있습니다.
- 주문과 결제의 상태 전이를 서버 규칙으로 강제할 수 있습니다.
- 테스트를 기능 추가와 함께 확장하는 습관을 만들 수 있습니다.

# 구현 순서를 왜 고정하는가
이 장에서는 다음 순서를 따릅니다.

1. 상품과 사용자 기초 데이터
2. 인증과 세션
3. 장바구니
4. 주문 생성
5. 결제 확인과 주문 확정
6. 운영자 조회 API

이 순서를 지키는 이유는 뒤 기능이 앞 기능의 확정된 데이터를 기대하기 때문입니다. 예를 들어 주문은 장바구니를 전제로 하고, 결제는 주문을 전제로 하며, 운영자 조회는 이미 발생한 이벤트들을 읽어야 합니다.

# 각 구현 단계의 핵심 질문
## 상품과 사용자 기초 데이터
- 어떤 상품 정보가 구매 흐름에 반드시 필요한가
- 사용자 역할은 어디서 구분되는가

## 인증과 세션
- 로그인 성공 이후 무엇이 사용자 식별 기준이 되는가
- 운영자와 일반 사용자 권한은 어떻게 나뉘는가

## 장바구니
- 같은 사용자의 장바구니는 어떻게 보존되는가
- 수량 변경과 금액 계산은 어디서 수행되는가

## 주문 생성
- 주문은 장바구니의 어떤 정보를 스냅샷으로 가져오는가
- 주문 생성 시점에 무엇이 다시 검증되어야 하는가

## 결제 확인
- 결제 시도와 주문 확정을 같은 단계로 볼 것인가
- 실패한 결제는 어떤 상태로 남겨야 하는가

# 이 장의 결과물
- 구매자 기준 핵심 API 세트
- 인증과 권한 처리
- 장바구니 보존 로직
- 주문 생성과 상태 전이
- mock payment 확인 흐름
- 운영자 조회용 기초 endpoint

# 이 장에서 흔히 하는 실수
- 주문보다 먼저 결제 연동부터 붙인다.
- 장바구니와 주문을 같은 데이터 구조로 다룬다.
- 인증 없이 장바구니와 주문 흐름을 임시로만 만든다.
- 실패 케이스 테스트를 생략한다.
- 운영자 API를 사용자 API와 같은 응답 구조로만 처리한다.

# 이 장에서 확인해야 할 체크리스트
- 로그인 사용자 기준으로 장바구니가 구분되는가
- 주문 생성 시 cart snapshot이 만들어지는가
- 결제 확인 전후로 주문 상태가 구분되는가
- 실패한 요청의 에러 이유가 명확한가
- 운영자가 주문과 결제 상태를 조회할 수 있는가

# 다음 장으로 넘어가기 전에
다음 장에서는 프론트엔드가 이 백엔드를 실제 사용자 경험으로 조직합니다. 따라서 이 장이 끝났을 때 백엔드는 단순히 응답을 주는 서버가 아니라, 프론트엔드가 신뢰할 수 있는 규칙과 상태를 제공하는 서버여야 합니다.
