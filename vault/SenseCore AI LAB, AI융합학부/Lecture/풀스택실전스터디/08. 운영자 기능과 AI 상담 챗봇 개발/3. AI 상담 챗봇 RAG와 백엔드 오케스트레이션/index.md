# 3. AI 상담 챗봇 RAG와 백엔드 오케스트레이션
이전 페이지에서는 운영자가 상품을 관리하는 도구를 만들었습니다. 이제 그 상품 정보와 주문 정보를 AI 상담 챗봇이 안전하게 참고할 수 있도록 백엔드 구조를 설계합니다. 이번 페이지의 목표는 "LLM을 호출한다"가 아니라, 상담 챗봇이 어떤 근거로 답하고 어떤 서비스 데이터에 접근할 수 있는지를 명확히 정하는 것입니다.

실전 서비스의 상담 챗봇은 자유 대화 장난감이 아닙니다. 상품 정보, 배송/교환/환불 정책, 주문 상태, 결제 상태처럼 실제 서비스 데이터에 기대어 답해야 합니다. 동시에 개인정보, 결제 정보, 운영자 전용 정보는 함부로 노출하면 안 됩니다. 그래서 이번 장에서는 RAG, 도메인 조회, 권한 검사, LLM 호출을 하나의 애플리케이션 서비스로 묶어 오케스트레이션합니다.

즉 이번 페이지는 AI 챗봇을 "프론트 채팅창 뒤에 LLM API를 붙이는 기능"이 아니라, 이커머스 도메인 경계를 지키는 상담 백엔드로 만드는 단계입니다.

# 왜 챗봇 백엔드를 먼저 설계해야 하는가
챗봇 UI를 먼저 만들면 입력창과 말풍선은 빠르게 만들 수 있습니다. 하지만 백엔드 경계가 없으면 곧바로 문제가 생깁니다.

- 어떤 지식을 근거로 답해야 하는지 불명확하다
- 주문 상태를 어디까지 조회할 수 있는지 애매하다
- 로그인 사용자와 비회원 질문의 응답 범위가 섞인다
- LLM이 모르는 내용을 꾸며내도 막기 어렵다
- 상담 로그를 저장할지, 얼마나 저장할지 기준이 없다

따라서 이번 교재에서는 챗봇 UI보다 먼저 상담 백엔드의 책임과 데이터 흐름을 정합니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 챗봇이 답변에 사용할 지식 소스는 무엇인가
- RAG 검색과 도메인 조회는 어떤 순서로 실행되는가
- 주문 상태 질문은 어떤 권한으로 조회하는가
- LLM 프롬프트에는 어떤 정보를 넣고 빼야 하는가
- 대화 로그는 어디까지 저장하는가
- 실패, 모름, 핸드오프는 어떤 기준으로 반환하는가

# 1. 챗봇의 상담 범위를 먼저 제한한다
처음부터 모든 질문에 답하는 챗봇을 만들면 구현과 검증이 불가능해집니다. 이번 단계에서는 상담 범위를 아래 네 가지로 제한합니다.

- 상품 정보 안내
- 주문 상태 안내
- 결제 실패 또는 결제 대기 안내
- 교환, 환불, 배송 같은 정책 안내

이 범위를 벗어나는 질문은 억지로 답하지 않고, 운영자 상담으로 넘길 수 있다는 응답을 준비합니다.

# 2. 지식 소스는 무엇으로 나누는가
챗봇이 참고할 지식은 크게 두 종류입니다.

- 정적 지식: FAQ, 배송 정책, 교환/환불 정책, 상품 설명 문서
- 동적 데이터: 상품 현재 상태, 주문 상태, 결제 상태, 사용자별 주문 내역

정적 지식은 RAG 검색에 적합합니다. 동적 데이터는 RAG 문서로 복사하지 말고 도메인 서비스를 통해 조회하는 편이 안전합니다. 예를 들어 주문 상태는 매번 바뀌므로 벡터 문서에 넣기보다 `Order` 조회 API 또는 애플리케이션 서비스를 통해 가져와야 합니다.

# 3. RAG는 어떤 문제를 해결하는가
RAG는 LLM이 답변할 때 근거 문서를 함께 제공하는 방식입니다. 이번 프로젝트에서는 아래 질문에 유용합니다.

- 배송은 보통 얼마나 걸리나요
- 교환은 며칠 안에 가능한가요
- 특정 상품의 사용법이나 소재가 궁금해요
- 결제 실패 시 다시 시도할 수 있나요

반대로 아래 질문은 RAG만으로 답하면 안 됩니다.

- 내 주문이 지금 어디까지 처리됐나요
- 내가 방금 결제한 주문이 성공했나요
- 내 장바구니에 담긴 상품 재고가 있나요

이 질문들은 사용자 세션과 도메인 데이터를 확인해야 합니다.

# 4. 상담 요청의 입력 계약을 정한다
백엔드 챗봇 API는 최소한 아래 입력을 받습니다.

- 사용자 메시지
- 현재 화면 맥락
- 선택된 상품 ID 또는 주문 ID
- 로그인 사용자 정보
- 이전 대화 일부

현재 화면 맥락은 중요합니다. 상품 상세 화면에서 "이거 배송 얼마나 걸려요?"라고 물으면 "이거"가 현재 상품을 의미할 수 있습니다. 주문 상세 화면에서 "왜 아직 안 왔나요?"라고 물으면 현재 주문 상태와 연결될 수 있습니다.

# 5. 오케스트레이션 순서는 어떻게 잡는가
상담 백엔드는 하나의 함수에서 LLM을 바로 호출하지 않습니다. 아래 순서로 처리하는 편이 좋습니다.

1. 요청 사용자와 세션을 확인한다
2. 메시지 의도를 분류한다
3. 필요한 도메인 데이터를 조회한다
4. 필요한 정적 문서를 검색한다
5. 프롬프트에 넣을 근거를 구성한다
6. LLM을 호출한다
7. 응답을 검증하고 후처리한다
8. 상담 로그를 저장한다

이 순서를 지키면 "무엇을 근거로 답했는가"를 나중에 추적할 수 있습니다.

# 6. 의도 분류는 얼마나 복잡해야 하는가
MVP에서는 복잡한 머신러닝 분류기를 만들 필요가 없습니다. 규칙 기반으로 시작해도 충분합니다.

초기 의도는 아래 정도로 나눕니다.

- product_question
- order_status_question
- payment_question
- policy_question
- unsupported_question

중요한 것은 의도 분류가 완벽한가가 아니라, 어떤 데이터 접근이 필요한지 결정할 수 있는가입니다.

# 7. 주문 상태 질문은 어떻게 처리하는가
주문 상태 질문은 권한 검사가 핵심입니다. 사용자가 자신의 주문을 물어보는 경우에만 상세 상태를 알려줘야 합니다.

처리 순서는 아래와 같습니다.

1. 로그인 여부를 확인한다
2. 주문 ID가 있으면 해당 주문의 소유자를 확인한다
3. 주문 ID가 없으면 최근 주문 후보를 제한적으로 조회한다
4. 주문 상태와 결제 상태를 상담 응답 근거로 넣는다
5. 주소, 결제 수단 원문, 민감 정보는 프롬프트에 넣지 않는다

이렇게 해야 챗봇이 편해지는 대신 보안이 약해지는 문제를 막을 수 있습니다.

# 8. 상품 질문은 어떻게 처리하는가
상품 질문은 정적 문서와 현재 상품 데이터를 함께 사용합니다.

- 상품명, 가격, 판매 상태는 `Catalog`에서 조회한다
- 상품 설명과 FAQ는 RAG 검색 결과로 보강한다
- 품절 또는 판매 중지 상태는 응답에 반영한다
- 구매 불가능한 상품을 추천하지 않는다

상품 데이터와 RAG 문서가 충돌하면 현재 도메인 데이터를 우선합니다. 예를 들어 FAQ에는 판매 중이라고 되어 있어도 admin에서 판매 중지한 상품이면 챗봇은 판매 중지 상태로 안내해야 합니다.

# 9. 결제 질문은 어떻게 처리하는가
결제 질문은 사용자를 불안하게 만들기 쉽기 때문에 응답이 명확해야 합니다.

초기 MVP에서는 아래 정도만 다룹니다.

- 결제 대기 상태 안내
- 결제 실패 상태 안내
- 결제 성공 후 주문 완료 안내
- 다시 시도해야 하는 경우 안내
- 운영자 상담으로 넘겨야 하는 경우 안내

실제 카드 승인번호, 결제 수단 상세, PG 원문 오류는 사용자가 아닌 운영자 화면에서 다루는 편이 안전합니다.

# 10. 프롬프트에는 무엇을 넣어야 하는가
프롬프트는 모델에게 모든 데이터를 던지는 곳이 아닙니다. 필요한 근거만 정리해서 넣는 계약입니다.

이번 단계에서는 아래 정보를 넣습니다.

- 상담 역할과 말투 규칙
- 답변 가능한 범위
- 검색된 정책 문서 요약
- 조회된 상품 또는 주문 상태
- 모르면 모른다고 말해야 하는 규칙
- 운영자 상담으로 넘기는 기준

반대로 아래 정보는 넣지 않습니다.

- 비밀번호나 토큰
- 결제 수단 원문
- 내부 운영 메모
- 다른 사용자의 주문 정보
- 디버깅용 시스템 정보

# 11. LLM 응답은 어떤 형태로 받아야 하는가
프론트가 안정적으로 처리하려면 LLM 응답을 자유 텍스트 하나로만 받지 않는 편이 좋습니다. 백엔드 응답은 아래 구조를 가질 수 있습니다.

- answer: 사용자에게 보여줄 답변
- citations: 참고한 문서 또는 데이터 출처
- handoff_required: 운영자 연결 필요 여부
- suggested_actions: 사용자가 다음에 할 수 있는 행동
- confidence: 내부 판단용 신뢰 수준

이 구조가 있으면 UI는 답변만 보여주는 것을 넘어, 주문 상세 이동이나 고객센터 연결 같은 다음 행동을 자연스럽게 제공할 수 있습니다.

# 12. 상담 로그는 어디까지 저장하는가
상담 로그는 디버깅과 운영에 유용하지만 개인정보 위험도 있습니다. MVP에서는 아래 정도를 저장합니다.

- 사용자 ID
- 메시지 요약 또는 원문
- 의도 분류 결과
- 참조한 상품 ID 또는 주문 ID
- handoff 여부
- 생성 시각

민감 정보가 들어갈 수 있는 원문 저장은 정책을 먼저 정해야 합니다. 교재에서는 처음부터 무기한 전체 원문 저장을 기본값으로 두지 않는 편이 좋습니다.

# 13. 실패와 모름은 어떻게 처리하는가
챗봇은 모든 질문에 답하려고 하면 위험합니다. 아래 상황에서는 명확히 제한 응답을 해야 합니다.

- 근거 문서를 찾지 못한 경우
- 주문 소유자 확인에 실패한 경우
- 결제 상태가 불명확한 경우
- 상담 범위를 벗어난 경우
- LLM 호출이 실패한 경우

이때 "아마도"라고 꾸며내기보다, 확인 가능한 범위를 말하고 다음 행동을 안내해야 합니다.

# 14. 백엔드 모듈은 어떻게 나누는가
초기 구현에서는 아래 정도의 모듈 경계가 적절합니다.

- `support/application/chat_service.py`: 상담 유스케이스 오케스트레이션
- `support/domain/intent.py`: 상담 의도와 정책
- `support/infra/retriever.py`: RAG 문서 검색
- `support/infra/llm_client.py`: LLM 호출
- `support/infra/conversation_repo.py`: 상담 로그 저장
- `catalog/application`: 상품 조회
- `order/application`: 주문과 결제 상태 조회

핵심은 챗봇 서비스가 `Catalog`, `Order`, `Payment` 데이터를 직접 테이블에서 훔쳐보지 않게 하는 것입니다. 각 컨텍스트의 애플리케이션 서비스를 통해 필요한 정보만 받아야 합니다.

# 15. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 챗봇 API가 메시지를 받을 수 있다
- 의도 분류가 최소 범위로 동작한다
- 정책 문서 또는 FAQ 검색이 가능하다
- 상품 질문에 현재 상품 상태를 반영한다
- 주문 질문에 사용자 권한 검사를 적용한다
- LLM 응답이 구조화된 형태로 반환된다
- 실패와 handoff 응답이 구분된다
- 상담 로그가 최소한으로 남는다

즉 챗봇은 이제 실제 서비스 데이터와 정책 근거를 바탕으로 답해야 합니다.

# 16. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- 상담 요청 API
- 상담 애플리케이션 서비스
- 의도 분류 규칙
- RAG 검색 모듈
- LLM client wrapper
- 상품/주문 조회 adapter
- 구조화된 상담 응답 타입
- 상담 로그 저장소
- 권한 실패, 근거 없음, LLM 실패 테스트

이 산출물이 있어야 다음 페이지에서 챗 UI, 운영자 핸드오프, 안전장치를 안정적으로 붙일 수 있습니다.

# 17. 자주 하는 실수
- 프론트에서 LLM API를 직접 호출한다
- 주문 상태를 권한 확인 없이 챗봇에 넣는다
- RAG 문서와 현재 도메인 데이터가 충돌할 때 우선순위를 정하지 않는다
- LLM 응답을 자유 텍스트로만 받아 UI가 다음 행동을 제공하지 못한다
- 상담 로그에 민감 정보를 무기한 저장한다
- 모르는 질문에도 그럴듯하게 답하도록 방치한다
- 챗봇이 다른 컨텍스트의 테이블을 직접 조회하게 만든다

이 일곱 가지 실수는 AI 기능을 붙이는 순간 서비스 신뢰와 보안을 동시에 흔들 수 있습니다.

이번 페이지에서는 AI 상담 챗봇의 백엔드 구조를 설계했습니다. RAG는 정적 지식의 근거를 제공하고, 주문과 결제 같은 동적 정보는 도메인 서비스를 통해 조회해야 한다는 점을 확인했습니다. 또한 의도 분류, 권한 검사, 프롬프트 구성, 구조화 응답, 상담 로그까지 하나의 오케스트레이션 흐름으로 정리했습니다.
