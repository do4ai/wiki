# 1. 주문, 결제, 고객 지원 운영 화면
이제부터는 구매자 흐름 위에 운영 관점의 화면을 본격적으로 올립니다. `07장`까지에서 우리는 상품 탐색, 인증, 장바구니, 주문 생성, 결제 확인, 주문 완료 화면까지 구현했습니다. 하지만 실전 서비스는 구매자 화면만으로 끝나지 않습니다. 운영자는 현재 어떤 주문이 들어왔는지, 결제가 어디서 막혔는지, 어떤 고객이 어떤 상태에서 도움을 필요로 하는지 볼 수 있어야 합니다.

이번 페이지의 목적은 `admin`을 단순 관리자 로그인 앱에서 벗어나, 주문과 결제와 고객 지원 상태를 읽고 판단하는 운영 surface로 확장하는 것입니다. 여기서 중요한 것은 "기능이 많아 보이는 대시보드"가 아닙니다. 운영자가 지금 무엇을 먼저 봐야 하는지, 어떤 상태를 구분해야 하는지, 지원 맥락을 주문 흐름과 어떻게 연결해야 하는지를 정리하는 것입니다.

즉 이번 장은 구매가 끝난 뒤의 데이터를 운영자가 실제로 다룰 수 있는 첫 화면 체계를 만드는 단계입니다.

# 왜 주문, 결제, 고객 지원을 같이 다루는가
운영자 입장에서 이 세 가지는 분리된 모듈이 아닙니다.

- 주문은 무엇이 구매되었는가를 보여준다
- 결제는 그 주문이 금액 기준으로 어떻게 처리되었는가를 보여준다
- 고객 지원은 사용자가 어디서 막혔고 어떤 도움을 원하는지를 보여준다

이 셋을 따로 보면 맥락이 끊깁니다. 예를 들어 결제 실패만 보면 기술 문제처럼 보일 수 있지만, 고객 문의와 같이 보면 사용자 경험 문제라는 것을 바로 알 수 있습니다. 따라서 이번 교재에서는 운영 화면의 첫 묶음을 주문/결제/지원으로 같이 보는 것이 자연스럽습니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 운영자 대시보드는 무엇을 먼저 보여줘야 하는가
- 주문 목록과 상세는 어떤 관점으로 읽혀야 하는가
- 결제 상태는 어떻게 구분하고 강조해야 하는가
- 고객 지원 맥락은 어디에서 연결해야 하는가
- 운영자 화면의 상태 배지와 기본 행동은 어떤 기준을 가져야 하는가

# 1. 운영 화면의 첫 책임은 무엇인가
이번 단계의 운영 화면은 모든 관리 기능을 다 넣는 것이 아닙니다. 우선은 아래 책임이 먼저 성립해야 합니다.

- 현재 주문 상태를 빠르게 파악
- 결제 실패나 지연 상태를 확인
- 고객 지원이 필요한 건을 식별
- 상세 화면으로 들어가 원인을 추적

즉 운영 화면은 "편집 도구"보다 먼저 "판단 도구"로 작동해야 합니다.

# 2. 왜 대시보드를 먼저 두는가
운영자는 처음부터 주문 상세 한 건만 보지 않습니다. 보통은 전체 상황을 먼저 봅니다. 따라서 초기 운영 화면은 아래 질문에 답할 수 있어야 합니다.

- 오늘 몇 건의 주문이 들어왔는가
- 결제 실패는 몇 건인가
- 처리 대기 건은 무엇인가
- 고객 지원이 필요한 주문은 있는가

이 질문에 답할 수 있는 최소 요약이 있어야 운영자가 어디를 먼저 봐야 하는지 판단할 수 있습니다.

# 3. 대시보드에 무엇을 먼저 보여줘야 하는가
이번 단계에서는 아래 정도면 충분합니다.

- 총 주문 수
- 결제 대기 또는 실패 건수
- 최근 주문 목록 일부
- 고객 지원 필요 건수 또는 placeholder

중요한 것은 카드 수를 늘리는 것이 아니라, 운영 우선순위를 보여주는 것입니다.

# 4. 주문 목록 화면의 첫 책임은 무엇인가
주문 목록은 운영자가 전체 흐름을 따라가는 기본 화면입니다. 따라서 아래 정보가 우선입니다.

- 주문 번호
- 주문 사용자
- 주문 시각
- 주문 총액
- 주문 상태
- 결제 상태

이 여섯 가지가 있으면 운영자는 목록만 보고도 어떤 주문이 어디 단계에 있는지 대략 파악할 수 있습니다.

# 5. 주문 상세 화면은 무엇을 보여줘야 하는가
상세 화면은 목록에서 보이지 않는 맥락을 채워야 합니다. 이번 단계에서는 아래 정도가 필요합니다.

- 주문 항목 목록
- 수량과 금액
- 주문 상태 전이 정보
- 결제 상태와 확인 결과
- 고객 문의 또는 지원 맥락 자리

즉 상세 화면은 "이 주문이 왜 현재 상태인가"를 설명할 수 있어야 합니다.

# 6. 결제 상태는 왜 별도로 강조해야 하는가
운영자에게 결제 상태는 주문 상태와 비슷해 보여도 우선순위가 다를 수 있습니다. 예를 들어 주문은 `payment_pending`인데 결제는 `failed`라면, 운영자는 즉시 봐야 할 수 있습니다. 따라서 이번 교재에서는 결제 상태를 주문 상태 안에 묻지 않고, 별도 배지나 별도 컬럼으로 드러내는 편이 좋습니다.

운영자 화면에서 최소한 아래 구분은 명확해야 합니다.

- 결제 대기
- 결제 성공
- 결제 실패

이 세 가지가 바로 보이지 않으면 운영 우선순위가 흐려집니다.

# 7. 상태 배지는 어떤 역할을 해야 하는가
상태 배지는 장식이 아닙니다. 운영자가 한눈에 현재 상태를 읽기 위한 압축 표현입니다. 따라서 이번 단계에서는 아래 원칙이 좋습니다.

- 주문 상태와 결제 상태를 구분한다
- 색상과 라벨이 일관된다
- 목록과 상세에서 같은 의미를 유지한다

즉 같은 `failed`인데 어떤 화면에서는 빨갛고 어떤 화면에서는 회색이면 안 됩니다.

# 8. 고객 지원 맥락은 어디서 연결해야 하는가
이번 장은 아직 AI 챗봇 자체를 붙이는 장은 아니지만, 운영자 화면은 고객 지원 맥락이 들어올 자리를 먼저 가져야 합니다. 최소한 아래를 생각할 수 있어야 합니다.

- 이 주문과 관련된 문의가 있는가
- 최근 상담 요약이 있는가
- 운영자가 이후 어떤 조치를 해야 하는가

이 정보가 바로 다 완성되지 않더라도, 주문 상세 화면 안에서 지원 맥락을 볼 수 있는 구조는 먼저 잡아 두는 편이 좋습니다.

# 9. 목록에서 어디까지 액션을 열어야 하는가
초기 운영 화면에서는 목록에서 너무 많은 액션을 열기보다, 읽기와 이동 중심으로 두는 편이 좋습니다.

예를 들어 이번 단계에서는 아래 정도가 자연스럽습니다.

- 주문 상세로 이동
- 상태 필터
- 간단한 새로고침

반대로 모든 상태 변경 버튼을 목록 행마다 다 두기 시작하면 운영 화면이 금방 복잡해집니다.

# 10. 운영자 화면의 loading/empty/error는 어떻게 다뤄야 하는가
운영 화면도 사용자 화면만큼 상태 처리가 중요합니다.

- loading: 대시보드 요약과 주문 목록 로딩
- empty: 아직 주문이 없는 경우
- error: 조회 실패, 인증 실패, 권한 문제

특히 운영 화면에서는 empty 상태와 error 상태를 명확히 구분해야 합니다. 주문이 없는 것과 조회가 실패한 것은 완전히 다른 상황이기 때문입니다.

# 11. 권한 문제는 운영 화면에서 어떻게 보여줘야 하는가
운영 화면은 이미 admin surface이므로, 권한 문제를 더 명확히 다뤄야 합니다.

- 로그인 안 됨
- admin 권한 아님
- admin 권한은 있지만 조회 자체가 실패함

이 셋은 각각 다른 상태입니다. 따라서 보호 라우트와 조회 오류를 한 화면으로 뭉개지 않는 것이 중요합니다.

# 12. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 운영자 대시보드가 요약 정보를 보여준다
- 주문 목록이 렌더링된다
- 주문 상세에서 항목/금액/상태를 볼 수 있다
- 결제 상태가 별도 의미로 드러난다
- 고객 지원 맥락이 들어올 자리가 구조상 보인다

즉 운영자는 이제 실제 주문 흐름을 읽을 수 있어야 합니다.

# 13. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- 운영자 대시보드 화면
- 주문 목록 화면
- 주문 상세 화면
- 주문/결제 상태 배지 규칙
- 고객 지원 요약 또는 placeholder 영역
- 운영 화면 조회 API 연결

이 산출물이 있어야 다음 페이지에서 상품 운영 도구를 붙여 운영 surface를 더 확장할 수 있습니다.

# 14. 자주 하는 실수
- 대시보드를 지표 카드 모음으로만 만들고 실제 주문 맥락 연결을 빼먹는다
- 주문 상태와 결제 상태를 한 컬럼으로 뭉개서 보여준다
- 상세 화면에서 왜 이 주문이 현재 상태인지 설명할 정보가 부족하다
- 고객 지원 맥락을 완전히 별도 기능으로 보고 주문 화면과 연결하지 않는다
- loading/empty/error를 같은 화면 처리로 뭉갠다
- 목록에서 너무 많은 액션을 먼저 열어 화면을 복잡하게 만든다

이 여섯 가지 실수는 다음 장의 운영 도구와 AI 상담 챗봇 연동을 바로 불안정하게 만듭니다.

# 정리
이 페이지에서 우리는 운영자 surface를 주문과 결제와 고객 지원 상태를 읽는 화면으로 세우고, 대시보드와 목록과 상세가 각각 어떤 책임을 가져야 하는지 정리했습니다. 또한 운영자 화면은 단순 조회가 아니라, 어떤 주문을 먼저 보고 어떤 문제를 먼저 처리할지 판단할 수 있는 화면이어야 한다는 점도 확인했습니다.
