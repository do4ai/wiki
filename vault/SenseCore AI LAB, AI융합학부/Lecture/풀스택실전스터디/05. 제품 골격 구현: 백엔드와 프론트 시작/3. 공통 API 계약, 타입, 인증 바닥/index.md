# 3. 공통 API 계약, 타입, 인증 바닥
앞 페이지에서 백엔드 골격과 프론트 골격을 각각 세웠다면, 이제 두 축이 실제로 같은 말을 하도록 맞춰야 합니다. 이 단계가 빠지면 프로젝트는 겉으로는 풀스택처럼 보여도, 실제로는 프론트와 백엔드가 서로 다른 가정을 가진 채 따로 움직이게 됩니다.

풀스택 프로젝트에서 가장 흔한 초기 실패는 기능 부족이 아니라 계약 불일치입니다. 백엔드는 `snake_case`로 응답하고 프론트는 `camelCase`를 기대하거나, 어떤 API는 `{ data: ... }`를 주고 어떤 API는 바로 배열을 반환하거나, 로그인 성공 시 어떤 곳은 토큰을 주고 어떤 곳은 사용자 객체만 주는 식으로 구조가 흔들립니다. 이렇게 되면 이후 장에서 카탈로그, 인증, 장바구니, 주문 기능을 붙일수록 수정 비용이 계속 커집니다.

그래서 이번 페이지에서는 실제 기능 구현을 본격화하기 전에 아래 세 가지를 먼저 고정합니다.

- API 응답과 오류 응답의 공통 형식
- 프론트와 백엔드가 공유하는 타입과 상태 이름
- 인증 정보를 어떻게 주고받고 어디에서 해석할지에 대한 바닥

# 왜 이 페이지가 골격 단계에서 필요한가
많은 초급 튜토리얼은 API를 하나 만들고 화면 하나 만들면서 그때그때 응답 구조를 정합니다. 하지만 이번 교재는 DDD 기반 이커머스 서비스를 끝까지 완성하는 흐름입니다. 따라서 API 계약은 기능 뒤에 따라오는 것이 아니라, 기능을 안전하게 붙이기 위한 선행 조건입니다.

특히 이번 프로젝트에서는 아래 이유 때문에 계약을 먼저 맞춰야 합니다.

- `storefront`와 `admin`이 같은 서버를 본다
- 인증이 구매자와 운영자 흐름 모두에 영향을 준다
- 주문과 결제는 오류 형식이 불안정하면 디버깅이 매우 어려워진다
- 챗봇 기능도 결국 서버 응답 계약 위에 얹힌다

즉 이번 페이지는 문서 정리용이 아니라, 이후 구현 충돌을 줄이기 위한 구조 정리 단계입니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 기본 성공 응답은 어떤 형식을 쓸 것인가
- 기본 오류 응답은 어떤 형식을 쓸 것인가
- 어떤 타입을 공통으로 관리할 것인가
- 인증 정보는 어떤 경로로 전달할 것인가
- 프론트는 응답과 오류를 어디에서 해석할 것인가

# 1. 공통 API 계약은 왜 필요한가
공통 API 계약이 없으면 프론트는 각 엔드포인트를 특별 취급하게 됩니다. 그러면 다음 문제가 생깁니다.

- 목록 응답마다 파싱 방식이 다르다
- 에러 표시가 화면마다 달라진다
- 인증 실패 처리 흐름을 공통화하기 어렵다
- 타입 정의가 API별로 중복된다

이번 교재에서는 "모든 API가 완전히 똑같아야 한다"가 아니라, 최소한 공통 패턴은 있어야 한다는 관점이 중요합니다.

# 2. 성공 응답 형식은 어떻게 맞출까
이번 프로젝트에서는 아래와 같은 기준을 먼저 정하는 것이 좋습니다.

- 단건 조회 응답
- 목록 조회 응답
- 생성/수정/삭제 응답
- 인증 응답

예를 들면 성공 응답에서 아래 질문에 답할 수 있어야 합니다.

- 실제 데이터는 어느 필드에 들어가는가
- 메타 정보는 별도 필드가 있는가
- 목록 응답에서 total, page 같은 정보가 필요한가
- 생성 성공 시 무엇을 반환하는가

중요한 것은 개별 API를 예쁘게 만드는 것이 아니라, 프론트가 예측 가능한 방식으로 응답을 읽을 수 있게 하는 것입니다.

# 3. 목록 응답은 왜 특히 먼저 정해야 하는가
이커머스 서비스에서는 목록 응답이 매우 자주 등장합니다.

- 상품 목록
- 주문 목록
- 운영자 대시보드 요약 목록
- 챗봇 FAQ 후보 목록

이때 어떤 API는 배열만 주고, 어떤 API는 페이징 메타를 주고, 어떤 API는 키 이름도 다르면 프론트는 매번 별도 어댑터를 써야 합니다. 따라서 이번 교재에서는 목록 응답의 최소 공통 패턴을 먼저 정하는 것이 좋습니다.

예를 들어 아래 정도를 먼저 합의할 수 있습니다.

- `items`
- `total`
- `page`
- `page_size`

초기 MVP에서 실제 페이징을 아직 쓰지 않더라도, 확장 방향을 고려한 최소 형식을 생각해 두는 편이 좋습니다.

# 4. 오류 응답 형식은 왜 더 중요할 수 있는가
성공 응답은 화면이 보이면 당장은 넘어갈 수 있습니다. 하지만 오류 응답이 제각각이면 프론트는 실패 상황을 일관되게 처리할 수 없습니다.

예를 들어 아래가 섞이면 문제가 됩니다.

- 어떤 API는 문자열만 준다
- 어떤 API는 `{ detail: ... }`를 준다
- 어떤 API는 필드 오류 목록을 준다
- 어떤 API는 내부 traceback 일부가 그대로 노출된다

이번 교재에서는 최소한 아래를 공통화해야 합니다.

- 오류 코드
- 사용자 노출 메시지
- 개발자가 추적할 수 있는 세부 정보 또는 필드 정보

즉 오류 응답은 단순한 실패 메시지가 아니라, 프론트와 디버깅 흐름을 동시에 지탱하는 계약입니다.

# 5. 인증 실패와 일반 오류를 왜 구분해야 하는가
프론트 입장에서 인증 실패는 단순한 일반 오류가 아닙니다. 인증 실패가 발생하면 보통 아래 행동이 뒤따릅니다.

- 로그인 화면으로 이동한다
- 세션을 정리한다
- 보호 라우트를 막는다
- 사용자에게 다시 로그인하라고 안내한다

반면 일반 비즈니스 오류는 현재 화면에서 메시지를 보여주고 수정하게 하는 경우가 많습니다. 그래서 이번 교재에서는 인증 실패와 일반 도메인 오류를 응답 구조와 status code 차원에서 분리하는 것이 중요합니다.

# 6. 어떤 타입을 공통으로 관리해야 하는가
모든 타입을 공유 패키지로 옮길 필요는 없습니다. 하지만 아래 타입은 프론트와 백엔드가 다른 말을 하면 곧바로 문제가 됩니다.

- 사용자 기본 정보
- 상품 요약 정보
- 주문 요약 정보
- 장바구니 항목
- 결제 상태
- 오류 응답 형식
- 인증 세션 또는 인증 응답

이 타입들은 두 앱과 서버가 같은 이름과 같은 상태 값을 보도록 맞추는 편이 좋습니다.

# 7. 상태 이름은 왜 먼저 고정해야 하는가
이커머스 도메인에서는 상태 이름 하나가 코드 전반에 퍼집니다.

- `pending`
- `paid`
- `failed`
- `cancelled`
- `completed`

문제는 프론트와 백엔드가 각각 다른 문자열을 쓰기 시작하면, 화면 분기와 필터와 배지가 모두 어긋난다는 점입니다. 따라서 이번 교재에서는 주문 상태, 결제 상태, 인증 상태 같은 핵심 enum 성격의 값은 처음부터 공통 상수나 공통 타입으로 관리하는 편이 좋습니다.

# 8. 프론트는 응답을 어디에서 해석해야 하는가
프론트가 API 응답을 해석하는 위치도 중요합니다. 가장 피해야 할 방식은 컴포넌트마다 응답 구조를 직접 뜯는 것입니다.

예를 들어 아래 방식은 좋지 않습니다.

- 페이지 컴포넌트 안에서 `response.data.items`를 직접 해석
- 다른 페이지에서는 `response.items`를 해석
- 또 다른 페이지에서는 실패 시 `detail`만 읽음

이번 교재에서는 최소한 아래 중 하나로 통일하는 편이 좋습니다.

- API client 레이어에서 1차 정규화
- feature 단위 데이터 access 레이어에서 해석

즉 컴포넌트는 가능하면 "이미 해석된 데이터"를 받는 쪽이 낫습니다.

# 9. 백엔드는 계약을 어디에서 보장해야 하는가
백엔드에서 계약은 단순히 README에 적는 것으로 끝나면 안 됩니다. 실제로는 아래 지점에서 보장되어야 합니다.

- response schema
- serializer 또는 response model
- 공통 예외 처리기
- 테스트

즉 백엔드가 계약을 코드 레벨에서 고정하지 않으면, 문서에 있는 규칙은 쉽게 무너집니다.

# 10. 인증 바닥은 무엇부터 정해야 하는가
이번 교재에서 아직 완전한 인증 구현 장으로 들어가지는 않았지만, 골격 단계에서 최소한 아래는 먼저 정해야 합니다.

- 로그인 성공 시 무엇을 반환할 것인가
- 인증 정보는 쿠키로 갈 것인가 토큰 헤더로 갈 것인가
- 운영자와 구매자 인증 경로를 어떻게 나눌 것인가
- 인증 실패 시 어떤 응답을 줄 것인가

이 질문에 대한 답이 없으면 다음 장의 카탈로그와 인증 구현에서 프론트와 백엔드가 계속 충돌합니다.

# 11. 이번 교재에서 인증 전달 방식은 어떻게 보는가
정답은 팀과 환경에 따라 달라질 수 있지만, 이번 교재에서는 최소한 "한 가지 기준을 일관되게 쓰는 것"이 중요합니다.

예를 들어 토큰 기반이라면 아래를 먼저 고정해야 합니다.

- access token 위치
- refresh 전략 유무
- 프론트 저장 위치
- API client에서 헤더 주입 위치

세션 쿠키 기반이라면 아래가 중요합니다.

- 쿠키 설정
- 인증된 사용자 확인 endpoint
- CSRF 고려 지점

중요한 것은 방식 자체보다, 프론트와 백엔드가 같은 방식으로 구현된다는 점입니다.

# 12. 운영자와 구매자 인증은 왜 분리 사고가 필요한가
같은 로그인이라도 운영자와 구매자는 완전히 같은 권한 모델이 아닙니다. 따라서 아래를 먼저 구분해서 생각해야 합니다.

- 로그인 endpoint를 분리할지
- 같은 endpoint라도 role을 응답에 담을지
- 프론트에서 어떤 surface를 열어줄지
- 보호 라우트가 어떤 기준으로 동작할지

이 구분이 있어야 `admin`에서 일반 사용자 세션으로 진입하는 문제를 초기에 막을 수 있습니다.

# 13. 기본 API 계약 표는 어떤 수준이면 충분한가
이번 단계에서 모든 endpoint를 완성할 필요는 없습니다. 하지만 최소한 아래 정도는 표로 나와 있어야 합니다.

- `GET /products`
- `POST /auth/login`
- `GET /auth/me`
- `GET /admin/orders`
- `GET /health`

각 항목에 대해 아래를 적으면 충분합니다.

- 목적
- 요청 파라미터
- 성공 응답 형태
- 인증 필요 여부
- 실패 가능 유형

이 표가 있으면 다음 장의 실제 구현과 테스트가 훨씬 수월해집니다.

# 14. 타입 공유는 어디까지 해야 하는가
공유 타입은 유용하지만, 무턱대고 늘리면 오히려 결합이 심해집니다. 이번 교재에서는 아래 원칙이 적절합니다.

- 계약 성격이 강한 타입만 공유한다
- UI 전용 파생 타입은 각 프론트 앱에 둔다
- 내부 DB 모델과 API 응답 타입을 무조건 동일시하지 않는다

즉 공유 타입은 "외부 경계 계약" 중심이어야지, 서버 내부 구현 세부사항까지 끌고 오면 안 됩니다.

# 15. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 문서로 옮기면 최소한 아래 산출물이 있어야 합니다.

- 핵심 endpoint 계약 표
- 공통 응답/오류 타입 정의
- 주문 상태와 결제 상태 같은 공통 상수
- 인증 응답 형식 정의
- 프론트 API client의 기본 해석 규칙
- 백엔드 response model 또는 schema

이 산출물이 있어야 다음 페이지에서 첫 통합 실행을 할 때 "왜 여기서 깨졌는가"를 훨씬 빨리 파악할 수 있습니다.

# 16. 자주 하는 실수
- 엔드포인트마다 응답 구조를 제각각 만든다
- 오류 응답은 나중에 정리하겠다며 미룬다
- 주문 상태 문자열을 프론트와 백엔드가 다르게 쓴다
- 인증 실패와 일반 오류를 같은 방식으로 처리한다
- 컴포넌트마다 API 응답을 직접 해석한다
- 내부 DB 모델을 그대로 외부 응답 타입으로 노출한다

이 여섯 가지 실수는 기능이 늘수록 풀스택 개발 속도를 빠르게 떨어뜨립니다.

# 정리
이 페이지에서 우리는 프론트와 백엔드가 같은 계약을 보도록 성공 응답, 오류 응답, 공통 타입, 상태 이름, 인증 바닥을 먼저 맞추는 이유를 정리했습니다. 또한 API 계약은 문서만이 아니라 response schema, 공통 예외 처리, 프론트 API client 해석 규칙, 테스트까지 포함해 코드로 보장되어야 한다는 점도 확인했습니다.
