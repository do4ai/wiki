# 풀스택 실전 스터디 커리큘럼
이 책은 하나의 실전형 이커머스 서비스를 처음부터 끝까지 완성하기 위해 만든 교본입니다. 학생은 구매자용 화면 하나만 만드는 것이 아니라, 프론트엔드와 백엔드와 인프라를 각각 세팅하고, 도메인을 설계하고, 주문이 실제로 생성되고, 운영자가 그것을 확인할 수 있으며, 마지막에는 배포와 검증과 확장까지 연결된 하나의 제품을 손에 넣게 됩니다. 목표는 기술 스택을 조각으로 아는 것이 아니라, 실제 서비스가 완성되는 순서를 몸으로 이해하는 것입니다.
# 이 책은 무엇인가
- 이 책은 강의 노트를 모아둔 참고서가 아니라, 위에서 아래로 순서대로 따라가면 하나의 풀스택 이커머스 프로젝트를 완성하도록 설계한 단일 교재입니다.
- 이 책은 `storefront`, `backend`, `admin`, `infrastructure`를 서로 다른 과목으로 분리하지 않고, 하나의 제품 안에서 어떻게 연결되는지 보여줍니다.
- 이 책은 코드만 설명하지 않고 기능명세, 화면설계, DDD, 구현 순서, 검증, CI까지 실무형 흐름을 함께 다룹니다.
# 이 책의 목표
- 프론트엔드, 백엔드, 인프라를 각각 독립적으로 세팅하고 다시 하나로 연결할 수 있게 합니다.
- DDD 관점으로 이커머스 도메인을 분석하고, 상품, 사용자, 장바구니, 주문, 결제, 운영자 기능의 경계를 설명할 수 있게 합니다.
- 백엔드 기능을 순서대로 구현하면서 API, 데이터 모델, 상태 전이, 검증 규칙을 직접 만들 수 있게 합니다.
- 프론트엔드 구조를 순서대로 세우면서 화면, 상태, 라우팅, API 연동, 역할 분리를 제품 구조 안에서 설명할 수 있게 합니다.
- 운영자용 surface를 별도의 제품 시야로 이해하고, 구매자 화면과 다른 책임을 가진 UI와 API를 설계할 수 있게 합니다.
- 로컬 실행에서 끝나지 않고 runtime, 배포 구조, CI, 품질 확인까지 이어지는 개발 습관을 갖게 합니다.
# 이 책을 읽는 방법
- 처음부터 끝까지 순서대로 읽습니다. 뒤 단계는 앞 단계에서 만든 구조와 약속을 이미 알고 있다고 가정합니다.
- 세팅 단계는 세 개의 트랙으로 나뉘지만, 셋 중 하나만 읽는 책이 아닙니다. 프론트엔드 트랙, 백엔드 트랙, 인프라 트랙을 모두 통과해야 전체 프로젝트가 완성됩니다.
- 각 장에서는 먼저 왜 이 단계가 필요한지 이해하고, 그다음에 어떤 산출물을 만드는지 보고, 마지막에 코드를 작성합니다.
- 막히면 뒤로 건너뛰지 말고 바로 앞 단계의 약속과 경계를 다시 봅니다. 실전 프로젝트는 기술 부족보다 순서 붕괴 때문에 더 자주 무너집니다.
- 예제는 가능한 한 직접 실행하고, 화면 결과뿐 아니라 API 응답, 데이터 저장, 로그, 검증 결과까지 함께 확인합니다.
# 이 책의 읽는 순서
1. 프론트엔드 프로젝트 세팅 트랙
2. 백엔드 프로젝트 세팅 트랙
3. 인프라 프로젝트 세팅 트랙
4. DDD 기반 이커머스 기획과 경계 정의
5. 백엔드 개발 순서 따라가기
6. 프론트엔드 세팅 완료와 아키텍처 구축
7. 운영자와 관리자 surface 만들기
8. 인프라 runtime, 배포, CI 연결
9. 최종 통합, 검증, 확장
이 순서는 단순히 기술 분야별 목차가 아닙니다. 먼저 각 레이어가 혼자 서는 법을 익히고, 그다음 도메인을 정리하고, 그 위에 백엔드를 먼저 세운 뒤 프론트엔드를 붙이며, 마지막에 운영과 배포를 닫는 순서입니다. 이렇게 해야 학생이 지금 만드는 코드가 제품의 어느 층에 속하는지 계속 잃지 않습니다.

# 각 단계에서 무엇을 배우는가
## 1단계. 프론트엔드 프로젝트 세팅 트랙
- 이 단계에서는 구매자용 서비스와 운영자용 서비스가 왜 분리되어야 하는지부터 이해합니다.
- React, TypeScript, 라우팅, 폴더 구조, 환경변수, API 클라이언트, 공통 UI 레이어를 어떻게 시작해야 나중에 화면이 많아져도 무너지지 않는지 배웁니다.
- 학생은 이 단계에서 화면을 많이 만드는 것이 아니라, 앞으로 붙일 백엔드와 운영자 기능을 견딜 수 있는 프론트엔드 기반을 먼저 세웁니다.
## 2단계. 백엔드 프로젝트 세팅 트랙
- 이 단계에서는 서버가 무엇을 책임지고 무엇을 책임지지 않는지 분명하게 잡습니다.
- FastAPI 애플리케이션 구조, 라우터 분리, 설정 관리, 데이터베이스 연결, ORM 모델 기본 틀, 인증 기초, 예외 처리, 테스트 시작점을 세웁니다.
- 학생은 이 단계에서 단순 CRUD 서버를 만드는 것이 아니라, 이후의 장바구니, 주문, 결제, 운영자 권한이 자연스럽게 들어갈 수 있는 백엔드 바닥을 만듭니다.
## 3단계. 인프라 프로젝트 세팅 트랙
- 이 단계에서는 로컬 개발 환경과 실행 환경이 프로젝트의 일부라는 사실을 배웁니다.
- 컨테이너, 환경변수 파일, 데이터 저장 위치, 네트워크 경로, 서비스 간 연결, 기본 실행 스크립트, 헬스체크를 준비합니다.
- 학생은 인프라를 나중 부가작업으로 미루지 않고, 처음부터 실행 가능성과 재현 가능성을 제품 설계 안에 넣는 습관을 갖게 됩니다.
## 4단계. DDD 기반 이커머스 기획과 경계 정의
- 이 단계에서는 이커머스를 화면 목록이 아니라 도메인 문제로 읽습니다.
- 상품, 사용자, 인증, 장바구니, 주문, 결제, 운영자 기능을 한 덩어리로 보지 않고 bounded context와 책임의 경계로 나눕니다.
- 기능명세서와 화면설계서를 읽고, 어떤 유스케이스가 어떤 aggregate와 규칙을 필요로 하는지 정리합니다.
- 학생은 이 단계에서 "무엇을 만들까"를 넘어서 "무엇을 어디까지 하나의 책임으로 둘까"를 배우게 됩니다. 이 단계가 약하면 뒤 구현은 빠르게 되더라도 계속 다시 뜯게 됩니다.
## 5단계. 백엔드 개발 순서 따라가기
- 이 단계는 이 책의 중심 축입니다. 백엔드는 화면보다 먼저 안정된 규칙을 가져야 하므로, 구현 순서 역시 의도적으로 고정합니다.
- 먼저 catalog와 사용자 기초 데이터를 읽고 쓸 수 있게 만들고, 그다음 인증과 권한을 붙입니다.
- 그 다음 장바구니를 구현하며 상품 선택과 수량 변경, 금액 계산, 사용자별 상태 보존을 다룹니다.
- 그 다음 주문을 만들며 cart snapshot, 주문 항목, 총액 계산, 상태 전이, 예외 조건을 설계합니다.
- 마지막으로 mock payment와 주문 확정을 연결하여 결제 확인이 주문 상태를 어떻게 바꾸는지 구현합니다.
- 이 단계에서 학생은 API를 하나씩 추가하는 것이 아니라, 도메인 규칙이 점점 두꺼워지는 과정을 경험합니다.
## 6단계. 프론트엔드 세팅 완료와 아키텍처 구축
- 백엔드가 기본 흐름을 제공하기 시작하면, 프론트엔드는 그 위에 사용자 경험을 조직합니다.
- 이 단계에서는 라우팅 구조, 레이아웃 체계, 페이지와 컴포넌트의 경계, 서버 상태와 로컬 상태의 분리, 인증 세션 처리, 폼 처리, 에러 피드백을 순서대로 잡습니다.
- 먼저 storefront의 뼈대를 세우고, 상품 목록과 상세, 장바구니, 체크아웃, 주문 완료 흐름을 붙입니다.
- 그 과정에서 API 호출을 화면 안에 흩뿌리지 않고, 재사용 가능한 데이터 접근 계층과 UI 구조를 만드는 법을 배웁니다.
- 학생은 이 단계에서 예쁜 화면을 만드는 것보다, 요구사항이 늘어나도 버틸 수 있는 프론트엔드 아키텍처를 먼저 배우게 됩니다.
## 7단계. 운영자와 관리자 surface 만들기
- 이 단계에서는 운영자 화면을 구매자 화면의 복사판으로 만들지 않습니다.
- 운영자는 상품과 주문과 결제 시도를 보고, 상태를 확인하고, 시스템 운영에 필요한 정보를 읽어야 하므로 정보 구조와 권한 구조가 다릅니다.
- 학생은 admin 로그인, 보호 라우팅, 운영자 API 연동, 대시보드, 주문 조회, 상품 관리, 운영 관찰 화면을 순서대로 만듭니다.
- 이 단계에서 중요한 학습은 화면 수를 늘리는 것이 아니라, 같은 백엔드 위에 서로 다른 역할의 제품 surface를 세우는 경험입니다.
## 8단계. 인프라 runtime, 배포, CI 연결
- 이 단계에서는 코드를 실행 가능한 제품으로 닫습니다.
- 로컬 개발용 실행과 배포용 실행의 차이를 이해하고, 컨테이너 구성, 서비스 연결, 정적 자원 서빙, API 경로, 비밀값 관리, 로그, 기본 관측 지점을 정리합니다.
- 그다음 테스트와 빌드와 린트와 검증 시나리오를 CI에 연결하여, 사람이 직접 보지 않아도 기본 품질이 꺾이지 않도록 만듭니다.
- 학생은 이 단계에서 배포를 버튼 누르기로 배우지 않고, 무엇이 runtime을 안정하게 만드는지 구조로 배우게 됩니다.
## 9단계. 최종 통합, 검증, 확장
- 마지막 단계에서는 이제까지 만든 것을 하나의 사용자 시나리오로 다시 묶습니다.
- 비회원과 회원, 구매자와 관리자, 성공 흐름과 실패 흐름, 주문 전과 주문 후 상태를 처음부터 끝까지 점검합니다.
- 그다음 SQLite에서 더 큰 데이터베이스로 옮기는 길, 외부 결제로 확장하는 길, 재고와 쿠폰과 배송을 추가하는 길, 모놀리식에서 더 큰 구조로 나누는 길을 설계합니다.
- 학생은 이 단계에서 프로젝트를 끝내는 법과 다음 프로젝트로 확장하는 법을 함께 배웁니다.
# 모든 수업이 따르는 공통 순서
- 먼저 이번 장이 제품 전체에서 어디에 놓이는지 설명합니다.
- 그다음 이 장에서 다루는 사용자, 운영자, 서버, 인프라의 책임을 나눕니다.
- 그 다음 필요한 도메인 규칙과 화면 또는 API 약속을 정리합니다.
- 그 다음에 코드 구조와 구현 순서를 제시합니다.
- 마지막에 실행, 검증, 다음 단계와의 연결로 마무리합니다.
이 책은 모든 장을 이 순서로 진행합니다. 그래서 학생은 언제나 지금 배우는 기술 자체보다, 그것이 제품의 어떤 층과 어떤 흐름에 들어가는지를 먼저 보게 됩니다.
# 이 책이 두는 약속과 가정
- 이 책은 학생이 처음부터 모든 기술을 알고 있다고 가정하지 않습니다. 대신 기본 문법은 안다고 가정하고, 제품을 완성하는 순서를 모른다고 가정합니다.
- 이 책은 프론트엔드, 백엔드, 인프라를 따로 떼어 암기시키지 않고, 서로 연결되는 지점에서 반복해서 설명합니다.
- 정의와 패턴은 실제 필요가 생긴 뒤에 나옵니다. DDD, 권한, 배포, CI도 먼저 이름을 외우게 하지 않고, 왜 필요한지 확인한 뒤 도입합니다.
- 이 책은 화면만 보이면 끝났다고 처리하지 않습니다. 항상 API, 데이터, 권한, 실행 환경, 검증까지 함께 본다고 약속합니다.
- 이 책은 실습이 순서대로 쌓인다는 가정 위에 쓰였습니다. 중간 장만 떼어 읽으면 개별 기술은 볼 수 있어도 제품 전체 흐름은 놓치게 됩니다.
# 이 책을 마치면
이 책을 끝내면 학생은 프론트엔드 하나, 백엔드 하나를 따로 만든 경험에서 멈추지 않고, 이커머스 제품 하나를 처음 세팅하는 순간부터 마지막 검증과 배포 직전까지 설명할 수 있어야 합니다. 더 중요한 목표는 기능 몇 개를 구현했다는 사실이 아니라, 어떤 순서로 프로젝트를 세워야 덜 무너지고, 어떤 경계를 먼저 정해야 나중에 확장할 수 있는지 아는 것입니다.

수강을 마친 뒤 학생은 구매자용 storefront, 관리자용 admin, 인증과 장바구니와 주문과 결제를 가진 backend, 그리고 이를 실행하고 검증하는 infrastructure를 하나의 프로젝트로 설계하고 구현할 수 있어야 합니다. 또한 새로운 기능이 들어와도 어디서부터 계획하고 어떤 층부터 수정해야 하는지 스스로 판단할 수 있어야 합니다.
