# 07. 운영자와 관리자 surface 만들기
이 장에서는 같은 백엔드 위에 두 번째 제품 surface를 세웁니다. 운영자 화면은 구매자 화면의 복사판이 아닙니다. 운영자는 구매를 하는 사람이 아니라, 시스템 상태를 읽고 판단하고 조치하는 사람입니다. 그래서 관리자 앱은 정보 구조와 권한 구조가 storefront와 달라야 합니다.

# 이 장의 목표
- 관리자 앱을 별도 제품으로 이해할 수 있습니다.
- 운영자 로그인과 보호 라우팅을 설계할 수 있습니다.
- 주문, 결제, 상품 데이터를 운영 관점으로 재구성할 수 있습니다.
- 읽기 중심의 dashboard와 관리 화면을 설계할 수 있습니다.

# 어떤 화면이 먼저 필요한가
- 로그인
- overview dashboard
- 주문 목록
- 상품 목록 또는 수정 진입점
- 결제 시도 및 주문 상태 관찰 화면

이 화면들은 구매자용 route와 목적이 다르므로 초기부터 별도 앱으로 유지하는 편이 안전합니다.

# 운영자 앱에서 중요한 것
운영자 앱은 시각 효과보다 정보 밀도와 판단 가능성이 중요합니다.

- 상태 배지
- 표와 필터
- 권한 없는 접근 차단
- 최근 이벤트 요약
- 실패 케이스 관찰

이 요소들은 나중에 붙이는 부가 UI가 아니라 운영자 제품의 핵심입니다.

# 이 장의 결과물
- 관리자 로그인과 세션
- 보호 라우팅
- 대시보드 개요 화면
- 주문 및 결제 목록
- 상품 관리 진입 화면

# 이 장에서 흔히 하는 실수
- 관리자 앱을 storefront UI 컴포넌트 복사로 처리한다.
- 운영자 권한을 프론트엔드 조건문으로만 막는다.
- 주문 상태와 결제 상태를 하나의 뱃지로 뭉갠다.
- 실패 흐름을 운영자가 볼 수 없게 만든다.
- 운영자 앱을 나중 부가 기능처럼 취급한다.

# 이 장에서 확인해야 할 체크리스트
- 관리자와 구매자 앱이 분리되어 있는가
- 관리자 라우트가 보호되고 있는가
- 주문과 결제 상태를 운영자가 읽을 수 있는가
- overview 화면이 핵심 운영 지표를 보여주는가
- 상품 관리와 주문 관찰의 책임이 분명한가

# 다음 장으로 넘어가기 전에
다음 장에서는 runtime과 배포와 CI를 연결합니다. 따라서 이 장이 끝났을 때 제품은 이제 사용자 앱과 관리자 앱을 모두 가진 상태여야 하며, 두 앱이 같은 백엔드를 공유하면서도 각자의 역할을 유지해야 합니다.
