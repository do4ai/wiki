# 08. 인프라 runtime, 배포, CI 연결
이 장에서는 지금까지 만든 코드를 실제로 실행 가능한 제품으로 닫습니다. 많은 학습 프로젝트가 코드 작성에서 멈추지만, 실전 프로젝트는 빌드, 테스트, 환경변수, 배포 경로, 품질 게이트까지 포함해야 완성에 가깝습니다. 이 장의 목적은 배포 버튼을 누르는 방법이 아니라, 무엇이 runtime을 안정하게 만드는지 이해하는 것입니다.

# 이 장의 목표
- 로컬 실행과 배포 실행의 차이를 설명할 수 있습니다.
- 프론트엔드와 백엔드의 배포 단위와 경로를 정리할 수 있습니다.
- 린트, 테스트, 빌드를 CI 파이프라인으로 연결할 수 있습니다.
- 운영 환경에서 필요한 기본 로그와 점검 지점을 설명할 수 있습니다.

# 어떤 것을 연결해야 하는가
- 프론트엔드 build
- 백엔드 테스트와 린트
- 공통 환경변수 검증
- 이미지 또는 실행 아티팩트 기준
- 배포 직전 확인 항목

이 연결이 있어야 사람이 직접 매번 확인하지 않아도 기본 품질이 유지됩니다.

# runtime을 볼 때의 관점
- 어떤 프로세스가 어떤 포트에서 뜨는가
- 프론트엔드는 어떤 API 경로를 바라보는가
- 정적 자원과 API 응답은 어디서 서빙되는가
- 시크릿은 어떤 방식으로 주입되는가
- 장애가 나면 어디서 먼저 확인하는가

이 질문을 놓치면 배포는 되어도 운영이 불가능해집니다.

# 이 장의 결과물
- CI 워크플로
- 빌드와 테스트 명령 정리
- 실행 환경별 설정 구분
- 배포 전 점검 항목
- 기본 운영 관찰 포인트

# 이 장에서 흔히 하는 실수
- 테스트와 빌드를 로컬에서만 확인한다.
- 환경변수 차이를 문서화하지 않는다.
- 프론트엔드와 백엔드 경로 규칙을 배포 단계에서 바꿔 버린다.
- 실패한 배포를 되돌릴 기준을 두지 않는다.
- 로그와 헬스체크를 나중에 붙인다.

# 이 장에서 확인해야 할 체크리스트
- 프론트엔드 build가 자동화되어 있는가
- 백엔드 테스트와 린트가 CI에 포함되어 있는가
- 환경변수와 시크릿 주입 방식이 분명한가
- runtime 점검 포인트를 설명할 수 있는가
- 배포 직전과 직후 확인 절차가 있는가
