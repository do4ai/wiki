# 2. 로컬 개발환경, compose, env
앞 페이지에서 저장소 구조를 고정했다면, 이제 그 구조를 실제로 돌릴 수 있어야 합니다. 실전형 풀스택 프로젝트에서 가장 먼저 무너지는 것 중 하나가 로컬 개발환경입니다. 폴더 구조는 멀쩡한데 누구는 `storefront`만 뜨고, 누구는 DB가 안 붙고, 누구는 환경변수가 없어서 챗봇 패널이 깨지는 식으로 서로 다른 상태가 생기기 쉽습니다.

이번 페이지의 목적은 이 차이를 없애는 것입니다. 어떤 개발자든 저장소를 받은 뒤 같은 명령으로 동일한 런타임을 띄우고, 같은 포트 규칙과 같은 서비스 이름으로 작업할 수 있어야 합니다. 그래야 프론트, 백엔드, 운영 화면, 챗봇 흐름을 같은 기준 위에서 동시에 개발할 수 있습니다.

# 왜 로컬 개발환경을 먼저 고정해야 하는가
구현보다 환경을 먼저 고정하는 이유는 간단합니다. 환경이 흔들리면 코드 문제와 실행 문제를 구분할 수 없기 때문입니다.

예를 들어:

- 어떤 사람은 `storefront`를 `5173`, 어떤 사람은 `3000`에 띄운다
- backend base URL이 사람마다 다르다
- admin은 켜지지만 API 포트가 달라 로그인만 실패한다
- DB는 로컬 설치에 의존하고, 다른 사람은 컨테이너로 띄운다

이 상태에서는 코드 리뷰도, 디버깅도, 강의 진행도 느려집니다. 따라서 이번 교재에서는 먼저 아래 세 가지를 고정합니다.

- 서비스 실행 방식
- 포트 규칙
- 환경변수 구조

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 어떤 서비스가 로컬에서 같이 떠야 하는가
- `compose`가 담당할 범위
- 앱별 포트 규칙
- 필수 `env` 항목
- 개발자가 따라야 할 실행 순서

다음 페이지의 DB, 시크릿, 실행 스크립트 설계도 이 문서 위에서 이어집니다.

# 1. 이번 교재에서 로컬에서 함께 떠야 하는 것
이번 이커머스 MVP에서 기본적으로 함께 떠야 하는 것은 아래 네 축입니다.

- `storefront`
- `admin`
- `server`
- `db`

선택적으로 아래가 붙을 수 있습니다.

- mock payment adapter
- 챗봇용 보조 서비스 또는 별도 worker

하지만 이번 교재의 첫 실행 기준은 "최소한 구매자 화면, 운영자 화면, 백엔드, 데이터 저장소가 한 번에 같은 방식으로 뜬다"입니다.

# 2. compose는 어디까지 담당하는가
`compose`는 개발환경의 공통 기반을 맞추는 데 좋지만, 모든 것을 무조건 컨테이너에 넣어야 하는 것은 아닙니다.

이번 교재에서는 아래처럼 나누는 것이 자연스럽습니다.

## compose가 맡기 좋은 것
- DB
- backend 런타임
- 필요한 보조 서비스
- 공통 네트워크와 포트 바인딩

## 로컬 프로세스로 띄워도 되는 것
- `storefront` dev server
- `admin` dev server

왜냐하면 프론트는 HMR과 디버깅 편의성이 중요하기 때문입니다. 물론 팀 기준에 따라 전부 컨테이너로 묶을 수 있지만, 학습과 개발 편의성 측면에서는 프론트는 로컬, backend와 DB는 `compose`로 두는 방식도 충분히 실용적입니다.

다만 이번 교재에서는 "한 명령으로 모두 띄우는 경험"도 중요하므로, 설명상으로는 `compose.yml`이 최소한 backend와 DB를 표준화하고, 필요하면 프론트도 같이 띄울 수 있는 구조를 권장합니다.

# 3. 포트 규칙을 먼저 정한다
포트는 사소해 보여도 팀 혼선의 원인이 됩니다. 처음부터 고정하는 편이 좋습니다.

예를 들어 이번 교재에서는 아래처럼 정할 수 있습니다.

| 서비스 | 권장 포트 | 이유 |
| --- | --- | --- |
| storefront | `5173` | Vite 기본 개발 포트와 자연스럽다 |
| admin | `5174` | storefront와 나란히 구분 가능하다 |
| server API | `8000` | Python/FastAPI 계열과 잘 맞는다 |
| db | `3306` 또는 `5432` | 선택 DB 기본 포트를 따른다 |

중요한 것은 특정 숫자 자체보다, 모두가 같은 숫자를 쓰는 것입니다.

# 4. 서비스 이름도 고정해야 한다
포트만큼 중요한 것이 서비스 이름입니다. `compose` 안에서 서비스 이름과 코드 안의 호스트명이 다르면 혼선이 생깁니다.

이번 교재에서는 아래처럼 단순하게 두는 편이 좋습니다.

- `storefront`
- `admin`
- `server`
- `db`

이 이름은 아래 위치에서 반복 사용됩니다.

- `compose.yml`
- `.env`
- 앱의 API base URL
- 문서의 실행 설명

즉 서비스 이름은 사람이 읽는 이름이 아니라, 팀 전체가 공유하는 런타임 식별자입니다.

# 5. env는 어떻게 나눠야 하는가
환경변수 설계의 핵심은 "무엇을 숨길 것인가"보다 먼저 "무엇을 어디에서 읽을 것인가"입니다.

이번 교재에서는 최소한 아래 구분이 필요합니다.

## 루트 공통 env
공통 런타임 설정이나 compose 변수

예:

- DB 호스트
- DB 포트
- 기본 API URL

## frontend 앱별 env
`storefront`와 `admin`이 읽는 값

예:

- API base URL
- 챗봇 활성화 플래그

## backend env
`server`가 읽는 값

예:

- DB 연결 정보
- secret key
- payment mock mode
- chatbot feature flag

중요한 점은 프론트와 백엔드가 같은 `.env` 파일을 아무 생각 없이 공유하지 않는 것입니다. 앱별 책임에 따라 읽는 값이 달라야 합니다.

# 6. 이번 교재에서 필요한 최소 env 항목
이번 MVP 기준으로는 아래 정도면 충분합니다.

## storefront/admin 쪽
- `VITE_API_BASE_URL`
- `VITE_CHATBOT_ENABLED`

## server 쪽
- `APP_ENV`
- `APP_HOST`
- `APP_PORT`
- `DB_HOST`
- `DB_PORT`
- `DB_USER`
- `DB_PASSWORD`
- `DB_NAME`
- `SECRET_KEY`
- `PAYMENT_MODE`
- `CHATBOT_ENABLED`

이 변수들을 너무 많이 늘리기보다, 실제 런타임에 필요한 것만 유지하는 편이 좋습니다.

# 7. 어떤 값은 env로 두고 어떤 값은 코드에 두는가
환경변수는 많을수록 좋은 것이 아닙니다. 자주 바뀌지 않는 상수까지 env로 빼면 오히려 읽기 어려워집니다.

## env로 두기 좋은 것
- 포트
- 외부 호스트
- 비밀값
- 환경별로 달라지는 설정

## 코드에 남겨도 되는 것
- 앱 내부 상수
- 화면 카피
- 항상 같은 기본 정책값

즉 `env`는 "변할 가능성이 있는 실행 설정"을 위한 도구입니다.

# 8. `.env.example`은 왜 필요한가
실전 프로젝트에서 새로 들어온 사람이 가장 먼저 보는 파일 중 하나가 `.env.example`입니다. 이것이 없으면 실행 장벽이 급격히 높아집니다.

이번 교재에서도 `.env.example`은 반드시 있어야 합니다.

## 포함해야 할 것
- 필요한 변수 이름
- 기본 예시 값
- 필수 여부에 대한 간단한 설명

## 포함하면 안 되는 것
- 실제 비밀값
- 개인 로컬 경로

`.env.example`은 실행 문서의 일부라고 생각하면 됩니다.

# 9. 권장 실행 순서
개발자는 아래 순서로 로컬을 띄우면 됩니다.

1. `.env.example`을 기준으로 환경변수 파일 준비
2. `compose`로 backend와 DB 기동
3. `storefront`와 `admin` dev server 기동
4. 브라우저에서 기본 경로 확인

이 순서가 중요한 이유는, API와 DB가 먼저 안정적으로 떠 있어야 프론트의 로그인/주문/운영 흐름을 바로 확인할 수 있기 때문입니다.

# 10. 로컬 개발환경에서 꼭 확인해야 하는 것
단순히 프로세스가 떴다고 끝이 아닙니다. 최소한 아래를 확인해야 "환경이 준비됐다"고 볼 수 있습니다.

- `storefront` 첫 화면이 열린다
- `admin` 로그인 화면이 열린다
- backend health 또는 기본 API 응답이 온다
- DB 연결 오류가 없다

이 네 가지가 확인되면 그다음부터는 기능 개발 문제로 넘어갈 수 있습니다.

# 11. 프론트와 백엔드의 base URL은 어떻게 맞추는가
로컬에서 자주 터지는 문제 중 하나가 base URL mismatch입니다.

예를 들어:

- storefront는 `http://localhost:8000`을 보는데
- admin은 `http://127.0.0.1:8000`을 보고
- backend는 CORS 기준이 다르게 잡혀 있으면

불필요한 디버깅이 발생합니다.

따라서 이번 교재에서는 문서, env, 코드 예시 모두 같은 host 표기를 쓰는 것이 좋습니다. 보통 `localhost`로 통일하는 편이 설명하기 쉽습니다.

# 12. compose 설계에서 피해야 할 것
처음부터 너무 무거운 구성으로 가면 학습 난이도가 불필요하게 올라갑니다.

## 피해야 할 예
- 캐시, 메시지 큐, 워커, 리버스 프록시를 한 번에 다 올린다
- 로컬에서 필요 없는 운영용 서비스까지 전부 띄운다
- DB도 두 개, API도 두 개, 프론트도 여러 개를 동시에 띄운다

이번 교재의 목표는 "실전형이지만 따라갈 수 있는 구성"입니다. 따라서 최소 런타임부터 시작하고, 이후 장에서 확장하는 것이 맞습니다.

# 13. 실습용 compose 초안
이번 교재를 따라 하는 학습자는 아래 정도를 목표로 잡으면 충분합니다.

```yaml
services:
  server:
    ...
  db:
    ...
```

그리고 프론트는 아래처럼 별도 dev server로 붙여도 됩니다.

- `client/storefront`
- `client/admin`

또는 필요하면 `compose`에 프론트까지 포함해도 됩니다. 중요한 것은 방법이 하나로 고정되어 팀 전체가 같은 기준을 쓰는 것입니다.

# 14. 자주 하는 실수
- `.env` 없이 각자 로컬 상수로 실행한다
- 포트를 사람마다 다르게 쓴다
- 프론트와 백엔드 base URL을 통일하지 않는다
- `compose`에 너무 많은 서비스를 한 번에 넣는다
- `.env.example` 없이 말로만 실행 방법을 전달한다

이 다섯 가지 실수는 개발 초반부터 팀 속도를 크게 떨어뜨립니다.

# 정리
이 페이지에서 우리는 로컬 개발환경을 같은 기준으로 맞추기 위해 `compose`, `env`, 포트, 서비스 이름, 실행 순서를 정리했습니다. 이제 누가 저장소를 받아도 같은 방식으로 backend와 DB를 띄우고, 같은 기준으로 storefront와 admin을 실행할 수 있는 바닥이 생겼습니다. 다음 페이지에서는 이 환경 위에서 실제 데이터 저장과 실행을 뒷받침하는 DB, 시크릿, 실행 스크립트 구성을 더 구체적으로 정리합니다.
