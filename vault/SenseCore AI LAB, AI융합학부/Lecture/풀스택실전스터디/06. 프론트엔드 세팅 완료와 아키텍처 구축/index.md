# 06. 프론트엔드 세팅 완료와 아키텍처 구축
이 장에서는 1장에서 세운 프론트엔드 바닥 위에 실제 사용자 경험을 얹습니다. 중요한 것은 화면 수를 늘리는 것이 아니라, 백엔드에서 확정한 규칙을 무리 없이 소비할 수 있는 UI 구조를 만드는 것입니다. 프론트엔드는 서버가 완성된 뒤에 꾸미는 층이 아니라, 서버 규칙을 사람이 사용하는 흐름으로 번역하는 층입니다.

# 이 장의 목표
- storefront 앱의 화면 흐름을 제품 시나리오 기준으로 정리할 수 있습니다.
- 페이지, 컴포넌트, API 접근 계층, 세션 처리를 역할별로 나눌 수 있습니다.
- 서버 상태와 로컬 UI 상태를 구분해서 다룰 수 있습니다.
- 장바구니와 주문 흐름을 사용자 관점에서 연결할 수 있습니다.

# 어떤 순서로 구현하는가
이 장에서는 보통 다음 순서를 따릅니다.

1. 앱 셸과 라우팅
2. 상품 목록과 상세
3. 로그인과 회원가입 진입
4. 장바구니
5. 체크아웃과 주문 생성
6. 주문 완료와 최근 주문 확인

이 순서는 화면 디자인 순서가 아니라 사용자 이동 순서입니다.

# 어떤 경계를 지켜야 하는가
- `app/`: 라우팅, provider, 인증 가드
- `lib/`: API 클라이언트, 타입, 세션 저장소
- `pages/` 또는 route 단위 화면: 실제 사용자 흐름
- `components/` 또는 feature 단위 UI: 반복되는 시각 요소

이 구분이 무너지면 프론트엔드는 곧바로 "아무 곳에서나 fetch하고 아무 곳에서나 세션을 읽는 구조"가 됩니다.

# 이 장의 결과물
- 구매자 storefront의 기본 route 구조
- 상품 조회와 상세 화면
- 로그인 이후 사용자 세션 반영
- 장바구니 수량 변경과 요약 표시
- 주문 생성과 결과 확인 흐름

# 이 장에서 흔히 하는 실수
- 화면 안에 API 호출과 데이터 가공을 모두 밀어 넣는다.
- 에러 상태와 로딩 상태를 뒤로 미룬다.
- 주문 생성 직전 검증을 프론트엔드만 믿는다.
- 서버 상태와 UI 임시 상태를 구분하지 않는다.
- 인증 전과 인증 후 흐름을 같은 화면 조건문으로만 처리한다.

# 이 장에서 확인해야 할 체크리스트
- route 구조가 구매자 여정 기준으로 설명 가능한가
- API 클라이언트 계층이 분리되어 있는가
- 세션과 사용자 정보가 앱 차원에서 관리되는가
- 장바구니와 주문 화면이 백엔드 상태와 일치하는가
- 실패 케이스에서 사용자에게 필요한 메시지가 보이는가

# 다음 장으로 넘어가기 전에
다음 장에서는 관리자 surface를 만듭니다. 따라서 이 장이 끝났을 때 storefront는 보기 좋은 데모가 아니라, 상품 조회부터 주문 생성까지 이어지는 실제 구매 흐름을 보여줄 수 있어야 합니다.
