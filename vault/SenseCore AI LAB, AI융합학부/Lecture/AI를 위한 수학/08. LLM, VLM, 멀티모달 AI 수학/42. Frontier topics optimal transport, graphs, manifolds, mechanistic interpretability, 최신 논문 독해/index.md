최신 AI 논문은 한 가지 수학만으로 읽히지 않는 경우가 많습니다. 분포를 옮기는 문제, 관계를 다루는 문제, 휘어진 공간을 다루는 문제, 모델 내부 회로를 해석하는 문제가 한 논문 안에 함께 등장하기도 합니다. 그래서 이 마지막 강의의 목표는 개별 기법을 전부 끝내는 것이 아니라, 새로운 주제를 읽을 때 어떤 질문 순서로 접근해야 하는지 몸에 익히는 것입니다. 이 강의는 곧 `"낯선 논문을 읽는 절차"`에 대한 실전 가이드입니다.

# 먼저 알아둘 말
- optimal transport: 한 분포의 질량을 다른 분포로 옮기는 최소 비용 문제다.
- graph: 노드와 간선으로 관계를 표현하는 구조다.
- manifold: 국소적으로는 평평하지만 전체적으로는 휘어진 공간이다.
- mechanistic interpretability: 모델 내부 계산을 회로나 알고리즘처럼 이해하려는 연구다.
- objective: 논문이 최소화하거나 최대화하려는 목표식이다.
- assumption: 논문이 성립하기 위해 미리 두는 가정이다.
- ablation: 구성 요소를 하나씩 제거하거나 바꿔 기여도를 보는 실험이다.
- appendix: 본문 뒤에 붙는 추가 유도, 실험 세부, 구현 조건이다.
- notation table: 논문에서 기호 의미를 한곳에 정리해 두는 표나 메모다.
- baseline: 비교 대상으로 쓰는 기존 방법이다.

# 이 강의에서 답할 질문
- 최신 논문에서 왜 서로 다른 수학 주제가 함께 나오는가?
- optimal transport, graph, manifold는 각각 어떤 상황에서 등장하는가?
- mechanistic interpretability는 어떤 질문을 던지는가?
- 낯선 논문을 읽을 때 어디서부터 읽어야 하는가?
- 수식이 길어도 어떻게 핵심 구조를 놓치지 않을 수 있는가?
- appendix와 ablation은 왜 반드시 봐야 하는가?

# 먼저 떠올릴 장면
- 처음 보는 논문을 펼쳤는데 수식과 약어가 빽빽하게 나와 있습니다.
- 이때 수식을 한 줄씩 처음부터 끝까지 번역하려 들면 바로 막히기 쉽습니다.
- 대신 `"이 논문은 무엇을 바꾸려 하는가, 데이터는 무엇인가, 손실은 어느 방향으로 움직이는가"`를 먼저 잡아야 합니다.
- 논문 독해는 수식을 많이 아는 것만이 아니라, 읽는 순서를 아는 기술이기도 합니다.

# 이 강의의 큰 그림
이 강의는 두 부분으로 읽으면 좋습니다.

1. 먼저 frontier topic에 자주 나오는 수학 도구들을 역할 중심으로 훑습니다.
2. 그다음 논문을 실제로 읽을 때의 체크리스트를 순서대로 정리합니다.

즉 이 강의는 `"도구 소개"`와 `"독해 절차"`를 연결하는 마무리 단계입니다.

# 먼저 버려야 할 생각
최신 논문을 읽을 때 가장 먼저 버려야 할 생각은 `"모든 수식을 한 번에 완전히 이해해야 한다"`는 압박입니다. 실제로는 먼저 논문이 무엇을 해결하려는지, 어떤 데이터와 어떤 목표식을 다루는지부터 잡는 편이 훨씬 중요합니다. 수학 도구는 그 목표를 달성하기 위해 중간에 선택된 장치일 뿐입니다.

즉 올바른 시작점은:

- `"무슨 문제를 푸는가?"`
- `"입력이 무엇인가?"`
- `"무엇을 최적화하는가?"`

이지, `"첫 페이지 수식을 모두 해석하자"`가 아닙니다.

# frontier topic 수학을 역할로 읽기
최신 논문에서 다양한 수학이 등장하는 이유는 간단합니다. 데이터와 문제 형태가 다양해졌기 때문입니다. 같은 생성모델 논문 안에서도:

- 분포 정렬 문제
- 구조적 관계 문제
- 저차원 기하 문제
- 내부 메커니즘 분석 문제

가 함께 섞일 수 있습니다.

중요한 것은 각 수학 분야의 엄밀 정의를 완벽히 외우는 것이 아니라 `"이 도구가 이 논문에서 무슨 역할로 들어왔는가"`를 구분하는 것입니다.

# optimal transport는 언제 나오는가
optimal transport는 두 분포를 얼마나 `"잘 맞출 수 있는지"`를 이동 비용 관점에서 보는 수학입니다. 생성모델, 도메인 적응, 분포 정렬 문제에서 자주 등장합니다. 핵심 식은 보통 다음처럼 적힙니다.

$$
W(P, Q)
=
\inf_{\pi \in \Pi(P, Q)}
\mathbb{E}_{(x,y)\sim \pi}[c(x,y)]
$$

처음 읽을 때는 이 식 전체를 외우기보다 `"두 분포를 연결하는 여러 방법 중 평균 이동 비용이 가장 작은 것을 찾는다"`는 뜻만 먼저 잡으면 충분합니다.

기호를 하나씩 풀면 더 분명합니다. 먼저

$$
P,\ Q
$$

는 맞추고 싶은 두 분포입니다. 다음으로

$$
\pi
$$

는

$$
P
$$

의 질량을

$$
Q
$$

의 질량으로 어떻게 옮길지 정하는 운송 계획입니다.

또

$$
\Pi(P,Q)
$$

는 양쪽 주변분포가 각각

$$
P,\ Q
$$

가 되도록 허용된 모든 운송 계획의 집합입니다. 마지막으로

$$
c(x,y)
$$

는 점

$$
x
$$

의 질량을 점

$$
y
$$

로 옮길 때 드는 비용입니다. 따라서

$$
\inf_{\pi\in\Pi(P,Q)}
$$

는 가능한 모든 운송 계획 중 평균 비용이 가장 작은 것을 고르라는 뜻입니다. 이 식을 이렇게 읽으면 optimal transport는 추상적인 거리가 아니라 `분포 하나를 다른 분포로 바꾸는 가장 싼 이동 계획`으로 이해할 수 있습니다.

논문에서 optimal transport가 나오면 먼저 다음 질문을 해야 합니다.

1. 어떤 두 분포를 맞추려 하는가?
2. 비용 함수

$$
c(x,y)
$$

는 무엇을 의미하는가?
3. 진짜 OT를 쓰는가, 아니면 근사 버전을 쓰는가?

즉 OT는 `"분포를 비교하고 맞추는 수학"`이라는 역할로 읽어야 합니다.

# graph는 언제 나오는가
graph는 데이터가 독립된 점들의 집합이 아니라 관계망일 때 등장합니다.

예:
- 사람과 사람의 연결
- 분자에서 원자와 결합
- 웹 페이지 링크
- 지식 그래프
- 장면 내 객체 관계

이런 경우 `"누가 누구와 연결되는가"`가 핵심 정보이므로 벡터만으로는 충분하지 않습니다. 논문에서 graph가 나오면 다음을 먼저 써 보면 좋습니다.

1. 노드는 무엇인가?
2. 간선은 무엇인가?
3. 관계 정보가 예측에 왜 중요한가?

즉 graph는 `"개별 점이 아니라 연결 구조 자체가 정보다"`라는 상황에서 등장합니다.

# manifold는 언제 나오는가
manifold는 고차원 데이터가 실제로는 더 낮은 차원의 구조 위에 놓여 있다는 생각입니다.

예를 들어 얼굴 이미지는 픽셀 차원으로 보면 매우 크지만, 실제 변화 요인은 조명, 자세, 표정처럼 훨씬 적을 수 있습니다. 즉 관측 차원은 크지만 실제 자유도는 더 작다고 보는 것입니다.

논문에서 manifold가 등장하면 다음을 물어야 합니다.

1. 데이터가 실제로 어떤 저차원 구조 위에 있다고 보는가?
2. 그 가정이 어떤 알고리즘 이득을 주는가?
3. 근처 이웃, 곡률, geodesic 같은 개념이 실제로 어디에 쓰이는가?

즉 manifold는 `"복잡한 고차원 데이터 뒤에 더 단순한 기하 구조가 있다"`는 가정의 역할로 읽으면 됩니다.

# mechanistic interpretability는 무엇을 묻는가
mechanistic interpretability는 모델을 블랙박스로 두지 않고, 내부 뉴런과 attention head, 회로가 실제로 어떤 계산을 하는지 밝히려는 연구입니다.

이 분야의 질문은 보통 다음과 같습니다.

- 이 모듈이 어떤 기능을 수행하는가?
- 특정 출력이 내부 어떤 경로를 통해 나왔는가?
- 어떤 head가 어떤 패턴을 감지하는가?
- 특정 회로를 지우면 어떤 능력이 사라지는가?

즉 성능 향상 자체보다 `"모델 내부 메커니즘 설명"`에 더 가까운 연구라고 보면 됩니다.

# 논문을 읽는 실제 순서
이제 도구 소개를 넘어서, 실제로 논문을 읽을 때 어떤 순서가 좋은지 정리합니다.

## 1. 문제를 한 문장으로 요약한다
가장 먼저 해야 할 일은 이 논문이 무엇을 더 잘하게 만들려 하는지 적는 것입니다.

예:
- 생성 품질 향상
- 도메인 차이 완화
- long-context 안정화
- multimodal grounding 개선
- 내부 회로 해석

이 문장을 못 적으면 뒤의 수식도 방향 없이 읽게 됩니다.

## 2. 입력과 출력을 적는다
논문을 읽을 때 의외로 가장 강력한 단계입니다.

- 입력은 무엇인가?
- 출력은 무엇인가?
- supervised인가, self-supervised인가?
- 분류인가, 생성인가, 정렬인가?

이 단계에서 데이터 타입을 먼저 적어 두면, 뒤에서 graph나 OT가 나와도 왜 나왔는지 훨씬 빨리 이해됩니다.

## 3. 변수 역할을 분리한다
새로운 기호가 나오면 먼저 이것이:

- 데이터인지
- 파라미터인지
- 분포인지
- 손실 항인지
- 보조 변수인지

를 적습니다.

수식이 어렵게 느껴지는 이유의 절반은 미분이나 적분 때문이 아니라, `"이 기호가 무엇을 뜻하는지"`가 안 잡혀 있기 때문입니다.

## 4. objective를 본다
그다음은 항상 objective입니다.

질문은 단순합니다.

1. 무엇을 최소화하거나 최대화하는가?
2. 각 항은 무엇을 보상하거나 벌점 주는가?
3. 기존 방법에 새로 추가된 항은 무엇인가?

objective를 읽을 때는 `"수식을 정확히 외우기"`보다 `"이 항의 역할이 무엇인가"`를 먼저 적는 것이 중요합니다.

## 5. baseline과 비교 포인트를 본다
좋은 논문은 기존 대비 무엇이 달라졌는지가 분명해야 합니다.

- 기존 baseline은 무엇인가?
- 바뀐 것은 구조인가, 손실인가, 데이터인가, 학습 절차인가?
- 성능 향상은 어디서 오는가?

이 비교를 놓치면 논문을 읽어도 `"그래서 뭐가 새롭지?"`가 남게 됩니다.

## 6. ablation을 본다
ablation은 논문의 주장을 검증하는 핵심 구간입니다.

예를 들어:
- 이 항을 빼면 성능이 얼마나 떨어지는가?
- 이 모듈을 바꾸면 결과가 유지되는가?
- 정말 주장한 원인 때문에 좋아진 것인가?

ablation을 보면 `"핵심 기여"`와 `"부수 효과"`를 구분할 수 있습니다.

## 7. appendix와 구현 조건을 본다
본문만 읽으면 큰 아이디어는 이해할 수 있어도, 세부 유도, 하이퍼파라미터, 실험 설정, 데이터 전처리, 학습 안정화 조건은 놓치기 쉽습니다. 실제로 재현하거나 응용하려면 appendix 확인이 필수입니다.

특히 다음을 확인해야 합니다.

- optimizer와 learning rate
- batch size
- 데이터 필터링
- 평가 프로토콜
- 추가 regularization
- 구현 디테일

실제로 재현 실패의 상당수는 아이디어가 아니라 디테일을 놓쳐서 생깁니다.

# 수식이 길 때 바로 써먹는 질문
긴 수식이 나오면 바로 다음 질문을 던지면 됩니다.

1. 이 식은 loss인가, bound인가, estimator인가?
2. 각 항은 보상인가, 벌점인가, 제약인가?
3. 새로 등장한 항은 기존 방법에 없던 어떤 역할을 하나?
4. 이 식이 데이터 분포, 구조 정보, 기하 구조 중 무엇을 반영하려는가?

이 질문만으로도 대부분의 수식은 `"번역 불가능한 벽"`에서 `"역할 있는 부품 묶음"`으로 바뀝니다.

# 논문 독해용 체크리스트
실제로 논문을 읽을 때는 아래 순서로 메모하면 좋습니다.

| 단계 | 적을 것 |
| --- | --- |
| 문제 | 무엇을 개선하려는가 |
| 데이터 | 입력, 출력, 모달리티, 구조 |
| 변수 | 데이터, 파라미터, 분포, 손실 항 구분 |
| objective | 무엇을 최적화하는가 |
| 새로움 | baseline 대비 무엇이 달라졌는가 |
| 증거 | ablation, 정량 결과, 실패 사례 |
| 재현성 | appendix, 구현 조건, 하이퍼파라미터 |

이 체크리스트를 반복하면, 낯선 논문도 읽는 속도와 품질이 눈에 띄게 좋아집니다.

# 왜 이 강의가 마지막에 오는가
이 강의가 마지막에 오는 이유는 분명합니다. 앞의 37~41강에서:

- attention
- pretraining
- alignment
- contrastive learning
- VLM 정보 흐름

을 이미 봤기 때문에, 이제 낯선 논문을 만나도 `"완전히 처음 보는 수학"`으로 느끼지 않고 `"어느 문제 맥락의 어떤 도구인가"`로 분류할 수 있게 되기 때문입니다.

즉 이 강의의 목표는 frontier topic을 전부 마스터하는 것이 아닙니다. 낯선 수학이 나와도 멈추지 않고, 문제 정의, 데이터 구조, 변수 역할, objective, 증거, 구현 조건의 순서로 읽어 내려갈 수 있게 만드는 것입니다.

# 자주 생기는 오해
## 오해 1. 논문은 수식을 처음부터 끝까지 다 이해해야 한다
핵심은 문제, 데이터, objective, 기여를 먼저 잡는 것입니다. 수식은 그다음입니다.

## 오해 2. 어려운 수학이 나오면 논문 전체를 못 읽는다
대부분은 도구 전체를 완벽히 아는 것보다 `"왜 여기 등장했는가"`를 먼저 아는 편이 더 중요합니다.

## 오해 3. appendix는 선택사항이다
재현과 응용을 생각하면 appendix는 필수입니다.

# 예제
1. 손실식을 읽는 첫 질문
문제: 새로운 논문에서 긴 손실식이 나오면 무엇부터 확인해야 하는가?
풀이: 무엇을 최소화하거나 최대화하는지, 각 기호가 데이터인지 파라미터인지, 각 항이 어떤 역할을 하는지부터 본다.
해설: 식을 암기하기보다 역할로 해석해야 전체 구조를 잃지 않는다.

2. manifold가 등장하는 이유
문제: 데이터가 manifold 위에 있다고 말하는 이유는 무엇인가?
풀이: 겉보기 차원은 크지만 실제 변화 요인은 더 적다고 보기 때문이다.
해설: 고차원 관측 뒤에 더 단순한 구조가 있다는 가정을 표현하는 말이다.

3. graph를 읽는 핵심 질문
문제: graph 관련 논문을 읽을 때 가장 먼저 적어야 하는 것은 무엇인가?
풀이: 노드가 무엇이고 간선이 무엇인지부터 적어야 한다.
해설: 그래프의 의미 단위를 모르면 뒤의 message passing 식도 해석이 흐려진다.

4. appendix를 보는 이유
문제: 논문 본문만 읽고 끝내면 놓치기 쉬운 것은 무엇인가?
풀이: 세부 유도, 하이퍼파라미터, 실험 설정, 구현 조건을 놓치기 쉽다.
해설: 재현성과 실제 적용 가능성을 보려면 appendix 확인이 필수다.

# 핵심 정리
- 최신 논문은 여러 수학 도구가 섞여 나오는 경우가 많다.
- 중요한 것은 도구의 완전한 엄밀성보다 `"이 논문에서 무슨 역할인가"`를 먼저 파악하는 것이다.
- optimal transport는 분포 정렬, graph는 관계 구조, manifold는 저차원 기하, mechanistic interpretability는 내부 회로 설명 문제로 읽을 수 있다.
- 논문 독해는 문제, 데이터, 변수, objective, baseline, ablation, appendix 순서로 읽는 것이 효과적이다.
- 수식은 암기 대상이 아니라 역할 분해 대상이다.
